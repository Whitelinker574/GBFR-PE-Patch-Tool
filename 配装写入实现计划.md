# 配装写入功能实现计划（LoadoutApply）

> 由 66-agent 设计工作流产出，经三视角评审 + 14 条对抗验证纠错。主干=安全优先(方案1)+同角色模板(方案2)。

## 一句话
复用 ProgressionApply 的「拒游戏运行 → 自动备份 → 原地改值 → FixChecksums → 回读逐字段验证」骨架，新增 LoadoutApply，把用户从"该存档已有资源"里拼出的一套配装（名称/武器/因子/技能/专精）原地写入指定预设槽（UnitID 20000..20614）；只引用存档里真实存在、且已被游戏判为合法的资源，绝不新造因子、绝不生成专精、绝不跨角色搬运，任何引用无法解析或归属存疑一律拒绝写入。

## 范围
【第一版做】(1) 写入/覆盖单个预设槽的六字段，字段值全部来自当前存档已存在的资源；(2) 复制整套已有配装到同角色另一槽（字节级搬运，最安全）；(3) 清空预设槽（3003=EmptyHash、名称清零、1403/1404/3007 复位，1402 保持原值）；(4) 配装的 JSON 导入/导出（用 hash 定位，跨档可移植，导入时在目标档解析回 SlotID）。

【第一版明确不做，因为依赖未验证假设，宁缺毋滥】(a) 不新造因子——PatchSigil 恒把 2706(装备角色) 写 EmptyHash，"预设是否要求被引用因子的 2706 指向该角色"尚无实测，造因子塞进预设可能写出游戏不认的状态；毕业配装若缺因子，请先用现有"因子生成"功能造好、再来引用。(b) 不生成/优化专精——技能盘拓扑、点数上限、与角色 MSP 余额的关系全部未验证，凭空生成一套专精可能非法；"理论最优"只能通过"从你另一套已存配装整体复制一份已被游戏接受的专精"来近似。(c) 不写队伍实时槽(104000+)与角色当前装备(10000+)，只碰玩家保存的预设。(d) 不跨角色放置角色专属因子/技能/专精。(e) 不碰 2701/2801 发号器（不造资源就不会引入撞号）。

## Go 接口
```go
新增 loadout_write.go（读侧 loadout.go 不动），沿用 ProgressionApply 形状：

// 写入闸门。changes 逐条独立全量预校验通过后才落盘；任一条失败即在 FixChecksums 前返回，存档文件不被触碰。
func (a *App) LoadoutApply(inputPath, outputPath string, changes []LoadoutWrite) (*LoadoutApplyResult, error)

// 只读：给编辑器提供"该角色可安全引用的资源池"，杜绝前端瞎猜。
func (a *App) LoadoutEditContext(path, charaHash string) (*LoadoutEditContext, error)

type LoadoutWrite struct {
  UnitID          uint32   // 目标预设槽 20000..20614；拒绝 104000+/10000+/非 15 对齐块外
  ExpectCharaHash string   // 前端认定的槽位归属角色；写前与"同块占用兄弟槽的 3003"比对，不符即拒绝（换档/并发保护）

  Name            string   // 配装名称 UTF-8，≤63 字节（留 NUL）

  SigilSlotIDs    []uint32 // 1403：≤12 个因子 SlotID(2702)，各须解析到现存非空因子，禁止重复 SlotID
  SkillHashes     []string // 1404：≤4 个技能 hash，须来自该角色现有配装
  MasteryHashes   []string // 3007：≤50 个技能盘节点 hash，逐字复制自该角色某已存配装，顺序原样保留
}

type LoadoutApplyResult struct {
  OutputPath, BackupPath      string
  SlotsWritten, SlotsCleared  int
  VerifiedFields              int   // 回读后逐字段命中的数量，用于成功文案"已写入并验证 N 项"
}

type LoadoutEditContext struct {
  CharaHash, CharaName, OwnerCode string  // OwnerCode=推导出的 PLxxxx，可能为空=无法确定
  BlockBase uint32                          // 该角色 15 槽的 UnitID 基址
  Slots     []LoadoutSlotInfo               // 15 槽：unitId/slot/occupied/name
  Weapons   []LoadoutPickWeapon             // 该角色专属 + 通用武器（含 slotId/hash/name/ownerCode）
  Sigils    []LoadoutPickSigil              // 可安全放置的因子：通用因子 + 已被该角色现有配装接受过的角色因子（含 slotId/hash/name/level/category）

func (e *unitEntry) Bytes() []byte
func (e *unitEntry) SetBytes(buf []byte) error   // 先把整块 ValueCnt 字节清零再拷贝，len(buf)>ValueCnt 报错

// SaveData 新增编排辅助：
func (s *SaveData) resolveBlockChara(unitID uint32) (charaHash uint32, ok bool, err error) // 从同块占用兄弟槽的 3003 推导角色，全空/冲突则 ok=false
func (s *SaveData) writeLoadoutName(unitID uint32, name string) error // 3002：UTF-8 编码+NUL 填满 64 字节缓冲，写前断言 ValueCnt==64
func (s *SaveData) writeLoadoutVector(idType, unitID uint32, vals []uint32, wantCnt int, pad uint32) error // 1403(13,pad=0)/1404(4,pad=EmptyHash)/3007(50,pad=EmptyHash)，逐个 SetUint32At，尾部补 pad，ValueCnt 不符即拒绝
func (s *SaveData) VerifyLoadout(w LoadoutWrite, charaHash uint32) (int, error) // 仿 VerifySigil，回读逐字段比对
```

## 合法性校验
写前对每条 LoadoutWrite 做完整预校验，全部通过才动缓冲区：

槽位：UnitID∈[20000,20614] 且是预设（loadoutSlotOf 返回 isParty=false）；六字段 3003/3002/1402/1403/1404/3007 都能 findUnit 命中，且实测 ValueCnt 恰为 1/64/1/13/4/50（不信任注释，按目标档现值断言，任一不符即拒绝）。

归属角色：resolveBlockChara(UnitID) 扫同块 15 槽 3003，取非空且一致的那个作为块角色 hash；块全空（无法确定角色）或多个非空互相冲突 → 拒绝；块角色 hash 必须等于 ExpectCharaHash，否则拒绝（防换档/前端算错块）。写入时 3003 写这个"实测推导值"，不盲信前端传入值。

名称(3002)：UTF-8 编码后字节数≤63；>63 拒绝（不截断，交回用户改）；写 64 字节缓冲=编码+其余全 0x00（覆盖空槽可能残留的旧名尾巴）。

武器(1402/WeaponSlotID)：SlotID 经 2802→武器槽 UnitID→2803 hash，hash 必须非空(≠EmptyHash≠0)；查目录得 ownerCode；ownerCode 为空(通用)放行；否则必须等于该角色的 PLxxxx。PLxxxx 由两源交叉推导——(i) 该角色现有配装 3007 节点经 skillboard_nodes.json 的 char 反查，(ii) 该角色现有配装武器的 ownerCode；两源须一致。若无法确定 PLxxxx，则只放行通用武器、拒绝任何带 ownerCode 的武器。

因子(1403/SigilSlotIDs)：≤12 个；各 SlotID 经 2702→因子槽 UnitID→2703 hash 必须解析到现存非空因子（复用 loadout.go 的 gemBySlotID 判空逻辑），解析不到→拒绝（绝不写悬空引用）；SlotID 不得重复（同一枚物理因子装两次，判非法）；因子 category==character_sigil 时须归属该角色——因 sigils.json 只有 GEEN_xxx 家族号、无 PLxxxx 字段（角色因子→角色映射未验证），v1 采白名单：只放行该 hash 已出现在该角色某现有配装/实时装备中的角色因子（游戏已亲自接受过=证明可装），否则拒绝。非角色因子(support/opus/special/null 共 96 个)视为通用放行。

技能(1404/SkillHashes)：≤4 个；每个 hash 须来自该角色现有配装的 1404 集合（技能同为角色相关，precedent 白名单，防跨角色）；尾部补 EmptyHash。

专精(3007/MasteryHashes)：≤50 个；整套须逐字来自该角色某一套已存配装（MasterySources），或保持目标槽原值；逐节点经 skillboard_nodes.json 校验 char==该角色 PLxxxx，任一不符→拒绝；顺序原样保留（节点是否要求特定顺序未验证，只复制不重排）；尾部补 EmptyHash。

clear：3003=EmptyHash、3002=64×0x00、1403=13×0、1404=4×EmptyHash、3007=50×EmptyHash；1402 保持原值（28/26/0 是随存档变的武器 SlotID，跨档硬写会指向错误武器，故不动）。

## 安全机制
备份：完全复用 SaveData.Write——覆盖同档时先建时间戳 .pre-edit-*.bak，并触发整组存档快照 autoSnapshotBeforeSaveWrite；BackupPath 回传前端展示。用户可另存到 outputPath 完全不碰原档。

写保护：samePath(input,output) 且 findProcessByName(charaProcessName) 命中（游戏在跑）→ 直接拒绝，避免游戏把内存旧数据回写覆盖。

失败回滚：所有校验在改缓冲区前完成，任一条不通过就在 FixChecksums/Write 之前 return err，磁盘文件从未被触碰（缓冲区仅内存态，丢弃即回滚）；采「先全量预校验、后统一写」两段式，即便内存态也不会留半写。

写后验证：Write 前 FixChecksums 重算哈希；写盘后 LoadSave(outputPath) 重新解析，VerifyLoadout 逐字段比对——3003==推导角色 hash、3002 的 64 字节==期望缓冲、1402==WeaponSlotID、1403 全 13 位(前 N 个==SlotIDs、其余==0、末位==0)、1404 全 4 位、3007 全 50 位全部精确命中，任一不符即返回"已写入但回读验证失败"的醒目错误（此时 .bak 已在，提示用户可还原）。VerifiedFields 计数回传。

残余风险的诚实兜底：即便结构与游戏亲写的配装同构，仍可能有未知的游戏侧校验；故成功文案明确提示"请进游戏读取该配装确认无异常，建议先在副本存档上试"。

## 风险与规避
 - 3007 专精节点顺序是否有语义未验证（20047 与 20048 前 4 位不同序）→ v1 只逐字复制已存配装、绝不重排，规避此风险，代价是不能生成新专精。
 - 角色因子→角色的映射未验证（sigils.json 只有 GEEN_xxx 家族号、无 PLxxxx 字段）→ v1 对角色因子采 precedent 白名单，可能漏掉玩家合法但从未在该角色配装里出现过的角色因子（假阴性，安全方向）；补一张 家族→PLxxxx 表是后续解禁前置。
 - charaHash→PLxxxx 推导可能失败（某角色配装无专精、且只用通用武器）→ 该角色武器校验降级为仅放行通用武器（安全方向，但限制了可选武器）。
 - 2706(被引用因子的装备角色) 是否需随预设指向该角色未验证 → v1 只引用游戏亲手造的现存因子(保留其原 2706)、不新造因子，把这条不确定挡在门外；这也意味着'一键毕业配装'需先自行造因子，v1 不提供。
 - '615 槽全预分配、ValueCnt 恒定'本地无 .dat 可独立复核 → 写入前对目标档实测每字段 ValueCnt，不符即拒绝，把假设风险转成可检测的拒绝。
 - 游戏遇悬空 SlotID 的行为项目内零记录 → v1 保证永不产生悬空引用（每个 SlotID 写前都验证解析到现存非空资源），从源头规避。
 - 即便结构与游戏亲写配装同构，仍可能存在未知游戏侧校验 → 强制备份+回读验证+明确提示用户先在副本档实机确认，把不可逆风险降到最低。
 - 把 3002 当 uint32 用 SetUint32At 写会越界冲毁后方 192 字节 → 新增 SetBytes 字节原语并断言 ValueCnt==64，配单测在真实存档形态上验证不越界。

# 已验证的关键事实（写入前提）

## 已验证的事实
- [fieldsem] 3002（配装名称）不在 UIntTable 而在 ByteTable 里，ValueCnt=64 的单位是「字节」不是「uint32」。全档 3002 恰好 615 条，每条 len(ValueData)=64 字节。
  证据：实测 ParseSaveData typed-table 扫描：`IDType 3002 出现在: map[Byte:615]`（其余 1402/1403/1404/3003/3007 均为 `map[UInt:...]`）；`>> Byte 3002 UnitID=20000 len=64`。对照 loadout.go:36 注释「3002=64」未区分元素宽度。
  影响：写名称必须按字节写 64 字节缓冲区，不能用 sigil_store.go 的 patchUint/SetUint32At（那是 4 字节步长，index i 会落到字节 4i，写 64 个 index 会越界冲掉后面 192 字节的相邻 FlatBuffer 数据）。需要新增一个「写 byte 向量」的 helper。
- [fieldsem] 名称「装备方案01」的实际字节是 UTF-8 无 BOM 明文：E8 A3 85 E5 A4 87 E6 96 B9 E6 A1 88 30 31（14 字节），其后 50 字节全部为 0x00，凑满 64 字节。
  证据：实测 dump：`>> Byte 3002 UnitID=20000 len=64 bytes=E8 A3 85 E5 A4 87 E6 96 B9 E6 A1 88 30 31 00 00 ...（共 50 个 00）`。UnitID=20048「装备方案04」为同样前 12 字节 + `30 34` + 50×00。
  影响：写名称 = UTF-8 编码 + NUL 填充到 64 字节。可用字节数上限为 63（须留终止符），中文一字 3 字节即最长 21 个汉字。写入前必须把整块 64 字节清零，否则会残留旧名尾巴。
- [fieldsem] 3002 未用部分的填充值是 0x00（纯零填充），不是 EmptyHash、不是空格。
  证据：实测：20000/20048 的 3002 第 14 字节起至第 63 字节全为 00；空槽 20051 的 3002 为 64 个 00 全零。

  更正：准确表述应为：**状态/toast 的上报**是子组件与外壳之间唯一的状态通道——所有工具组件（含写存档页）一律通过 emit('status', message, type)（type ∈ 'success'|'error'|'info'）上报，由 PatchTool 的 showStatus 显示在标题栏；因此新写的配装写入页应复用 emit('status', ...)，不要自建 toast（这条对存档安全无影响，可照做）。但壳↔子的"通信契约"并不唯一：还包括 (a) 向下的 props——如 HomeJournal 的 `:version`、MiscTools 的 `mode="stable|compatibility"`；(b) 其他事件——HomeJournal（外壳直接子组件）用 `emit('open', 工具id)`、`emit('warm', 工具id)` 做导航与预热，而非 status。设计写入页时若只需要"上报进度/成败"，用 emit('status', msg, 'success'|'error') 即可；但不要据此以为 status 是壳↔子之间唯一的接口。
