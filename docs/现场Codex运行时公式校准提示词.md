# GBFR 现场 Codex 运行时公式校准提示词

把下面“提示词正文”完整发给有游戏电脑上的 Codex。先解压交接包，再把 Codex 的工作目录设为包内的 `source`。

## 提示词正文

你正在一台已安装并能运行《碧蓝幻想：Relink》DLC 2.0.2 的 Windows 电脑上。当前目录是 `GBFR PE Patch Tool` 的完整源码。用户会保持游戏进程运行，并按你的短指令切换角色、装备、因子、专精、武器技能和召唤石。请直接完成现场诊断、代码修改、测试、构建和证据导出；不要只给分析建议，也不要把一长串人工 A/B 操作重新丢给用户。

### 当前基线

- 基线分支：`feature/loadout-fullscreen`
- 基线提交：`c302168e1bcaace018329514e7c94b2c2711977b`
- 当前生产程序 SHA-256：`025EC74B3A2118C122507B4DB6D8F3A29659FED88E328D2D2C0FBC493E68F873`
- 当前采样器只用 `PROCESS_QUERY_INFORMATION | PROCESS_VM_READ`，不写内存、不注入、不修改存档。
- 当前截图中的复现条件：工具选择“伊欧 · `4D0A60C3`”，游戏已经停在伊欧的装备/人物属性画面，肉眼可见的面板值约为攻击力 `17083`、暴击率 `124%`、昏厥值 `306`，但连接时报“游戏内尚无角色 4D0A60C3 的可用面板结果”。这些数字只用于寻找当时的运行时对象，禁止把它们硬编码为公式或固定偏移。
- 已确认的原始数据事实：伊欧 EX 专精节点 `1F52146F` 的动作值是 float32 `0.4`（位模式 `0x3ECCCCCD`），中文原文为“昏厥值+0.4”；运行时显示尺度尚未闭环。
- 专精存储结构有 `10/10/10/20 = 50` 个槽，但存档总 MSP 为 0 时，现有离线字段不能区分“专精系统未开放”和“已开放但未获得 MSP”。禁止再次把结构上限 50 当成当前解锁等级。

### 首要故障

先复现并修复“可见角色面板存在，但 `locateRuntimeCharacterPanelStatus` 找不到目标角色”的问题。关键代码：

- `runtime_character_panel.go`：固定 manager RVA、版本守卫、角色 ID 向量、状态 map、ready/eligibility、角色 hash 和最终四项面板偏移。
- `readonly_game_process.go`：只读进程连接和生命周期。
- `formula_sampler_app.go`：采样会话与状态对象捕获。
- `formula_sampler.go`：A1/B1/A2/B2 状态机。
- `formula_sampler_scan.go`：24 KiB 状态对象扫描和候选差分。
- `formula_sample_bundle.go`：脱敏证据包。
- `frontend/src/components/FormulaSampler.vue`：采样界面。

不要先猜一个新偏移。请做一个临时但结构化的“只读运行时对象枚举器”，在版本守卫通过后枚举 manager 的 ID 向量和状态 map，输出或在诊断界面显示：

1. ID 向量中的原始 ID；
2. map 节点 key；
3. ready、eligibility；
4. 状态对象中现有候选角色 hash；
5. 候选对象前 0x6000 字节内与屏幕 HP/攻击/暴击/昏厥四值相符的位置；
6. 选择角色的目录 hash 与运行时 ID/hash 的对应关系。

诊断输出可以在本机临时包含地址，但提交、ZIP 和截图中必须删除 PID、模块基址、绝对地址、用户名和本地路径。任何进入正式代码的定位逻辑必须由至少两个角色、每个角色三次稳定读取验证，不能只对伊欧硬编码。

重点检查以下假设，但要用证据淘汰，而不是直接认定：

- `4D0A60C3` 是目录/存档 hash，而运行时 map 的 key 或对象 hash 使用了另一套 ID；
- `runtimeCharacterPanelCharacterHashOffset = 0x59F0` 已变化或并非所有角色对象都填充；
- `ready == 1` 或 `eligibility != 0` 的过滤条件过严；
- 当前 manager 中存在多个状态对象，实际装备页对象不在现有 ID 向量；
- 游戏区域、队伍位置或控制角色导致对象生命周期不同；
- 版本守卫只证明几段代码相同，没有证明 manager/对象布局的全部偏移仍一致。

### 要实现的现场工作流

修复后把采样流程改成以程序自动观测为主：

1. 自动枚举当前可用角色状态对象，并把“目录角色名/目录 hash/运行时 ID”一起显示；用户只需确认当前屏幕角色，不应因为目录 hash 不等于运行时 key 而连接失败。
2. 连续显示游戏最终 HP、攻击力、暴击率、昏厥值，并标记每项的原始类型、相对偏移和连续读取稳定性。
3. 提供“开始记录变化/停止并分析”，在用户只切换一个配置时自动记录稳定前值、过渡期和稳定后值；程序自己判断 A/B 是否可重复。保留手动 A/B/A/B 作为严格验证模式，不再把它作为定位器工作的前提。
4. 对专精、防御力、伤害上限等未必改变四项面板的变量，自动对同一状态对象及其可证明关联的子对象做多轮差分。不得无限制扫描整个进程，也不得导出原始内存。
5. 在证据不足时显示“候选/负观察/尚未闭环”，禁止显示成“真实最终公式”。
6. 给每个已验证字段建立机器可读的版本化布局描述，至少包含游戏 EXE SHA-256、代码守卫、访问链、字段类型、相对偏移、样本角色数和证据等级。

### 本轮必须校准的内容

按下面顺序推进。每一步只让用户做一个很短的游戏操作，并由程序连续观测：

1. 当前角色最终 HP、攻击力、暴击率、昏厥值的真实回读。
2. 目录角色 hash 与运行时对象 ID 的映射。
3. 专精未开放、零 MSP、低等级专精和满级专精的真实边界；不能用 50 槽结构代替解锁状态。
4. 伊欧 `1F52146F` “昏厥值+0.4”的实际面板尺度。
5. 防御专精以及防御/支援联动：定位防御、伤害减免或相关候选字段。若最终效果只在受击结算出现，加入训练场“受击前 HP/受击后 HP/伤害值/当前 Buff”记录，不要拿四项人物面板无变化推断它无效。
6. 因子、武器基础值、武器技能、专精、角色上限突破、永久强化、召唤石主加护和副词条对最终数值的贡献。不可逆强化不能强行做同进程 A/B，应使用不同角色或成对存档，并明确证据边界。
7. 伤害上限和条件技能需要战斗/木桩样本时，单独记录，不得与城镇面板公式混为一谈。

### 工程与安全约束

- 诊断和公式研究阶段保持只读：仅允许 query-information 与 VM-read。不要注入 DLL、安装 hook、调用写内存接口或修改真实存档。
- 若确实需要写入构造 A/B，先在代码和报告中说明必要性，只使用用户明确指定的测试存档副本；本轮默认不做。
- 保留现有用户改动和未跟踪文件；禁止 `git reset --hard`、`git checkout --` 或删除工作区。
- 新建分支 `field/runtime-formula-calibration`，不要改写基线分支历史。
- 所有运行时定位修复先写失败测试，再实现。固定内存图测试必须覆盖：错误 hash 映射、过严 ready/eligibility、多个状态对象、损坏向量/map、地址溢出和版本守卫不匹配。
- 运行 `go test ./...`、`node --test frontend/src/*.test.js`、`npm run build`，然后生成新的 Windows amd64 EXE。
- 现场集成测试至少覆盖两个角色；记录游戏 EXE SHA-256、工具提交号和屏幕数值对照。截图必须来自实际应用窗口或游戏窗口，不能用错误裁切冒充应用结果。
- 不要为了让测试通过削弱版本守卫、地址边界、稳定读取或所有权令牌。

### 完成标准与交付

只有满足以下条件才算完成：

1. 当前截图场景可以连接，工具显示值与游戏屏幕逐项一致；
2. 换到第二个角色后自动识别且仍一致；
3. 退出装备页、切角色、重载存档后不会读到旧对象；
4. “昏厥值+0.4”和至少一个防御专精获得重复样本，或被诚实标记为仍缺哪一种证据；
5. 单元测试、前端测试、生产构建和启动检查通过；
6. 提交代码并生成以下交付物：
   - `field-calibration-report.md`：根因、改动、逐项对照、仍未知内容；
   - `field-runtime-evidence.zip`：只含脱敏后的布局、事件、候选和校验和；
   - `field-runtime-calibration.bundle`：包含现场分支提交的 Git bundle；
   - 新程序及其 SHA-256。

把以上交付物统一放到源码同级的 `RETURN-TO-MAIN-CODEX` 目录。最后告诉用户准确目录、提交号、EXE 哈希，以及还需要他在游戏里做的唯一下一步；如果没有下一步，明确说现场校准已经闭环。
