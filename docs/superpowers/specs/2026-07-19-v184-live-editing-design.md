# v1.8.4 新祝福与实时编辑安全升级设计

日期：2026-07-19

## 目标

在不复制第三方实现的前提下，借鉴 GBFR PE Patch Tool v1.8.3/v1.8.4 已公开的产品行为，为 GBFRDataTools 增加可关闭、可恢复、可验证的“背包祝福实时编辑”，同时修复现有因子实时编辑中同类的寄存器破坏风险，并补齐当前版本可确认的数据真值。

这次迭代只恢复安全、可验证的自动化：进程检测、记录捕获、失效提示和钩子清理可以自动执行；会改变游戏数值的写入仍由用户明确触发。不会自动重复应用旧补丁，也不会在指针过期后继续写。

## 上游结论与边界

- v1.8.3 新增“从背包选中祝福后读取、修改”的流程。
- v1.8.4 是针对该流程的崩溃热修：注入代码保存并恢复 `R10`，并补充显式关闭/恢复路径。
- 上游项目没有可见许可证。本项目只采用公开行为、记录布局和经过独立验证的事实，重新设计代码、测试与界面；不复制其 Go/Vue 源码。
- 上游的固定 RVA、逐字段写入和手动关闭仍有可改进空间。本项目继续沿用自己的整条记录事务写入、回读、回滚与存档快照机制。

## 后端设计

### 1. 安全代码洞

因子与祝福的代码洞都遵循同一约束：

1. 先保存临时使用的非易失/现场寄存器。
2. 将当前选中记录指针写入代码洞的数据区。
3. 恢复寄存器。
4. 原样执行被覆盖的游戏指令。
5. 跳回原执行流。

代码洞的拥有权必须通过完整前缀、内嵌数据地址、回跳地址与保存的原始字节共同验证。能识别旧版自有代码洞时，只允许安全恢复/卸载；新启用始终写入新格式。释放时先确认入口仍指向本工具的代码洞，再恢复原字节。远程页不立即释放，避免仍在执行的线程跳入已释放内存。

### 2. 祝福记录

一次捕获的是 24 字节记录：

| 偏移 | 字段 |
|---|---|
| `+0x00` | 第一词条 hash |
| `+0x04` | 第一词条等级 |
| `+0x08` | 第二词条 hash |
| `+0x0C` | 第二词条等级 |
| `+0x10` | 第三词条 hash |
| `+0x14` | 第三词条等级 |

更新流程：目录真值校验 → 再读代码洞指针并确认未改变 → 读取完整旧记录 → 创建实时存档快照 → 编码完整新记录 → 一次写入 → 写入回读 → 调用游戏保存函数 → 保存后回读。确定性失败时恢复完整旧记录、再次保存并验证恢复结果；如果远程保存线程超时而结果不确定，则不并发执行可能相撞的回滚，而是隔离当前游戏进程实例并要求重启游戏后再连接。成功后立即清空捕获指针，强制玩家重新选择下一条祝福。

### 3. 生命周期与状态

暴露 `GetOptions`、`Scan`、`GetStatus`、`Enable`、`Update`、`Disable` 六个入口。状态机为：

`未连接 → 已定位 → 等待选择 → 已捕获 → 写入中 → 写入成功/失败 → 等待重新选择`

页面挂载时只轮询状态；启用必须明确点击。离开页面、主动断开或应用退出都会调用 Disable/Detach 恢复入口指令。相同游戏进程上的重复连接应保持幂等，不能因为另一个页面查询连接而拆掉现有钩子。

#### 并发页面所有权

Wails 会并发派发前端调用，因此“先发出的 Acquire 一定先返回”并不成立。所有实时功能共享一条严格递增的 `requestID` 序列；后端在同一把进程互斥锁内拒绝 `0`、重复值和小于当前世代的请求，再签发不可伪造的 owner token。这个世代是跨页面、跨因子/祝福/超限/召唤石功能的全局世代，旧页面即使延迟返回也不能重新取得进程所有权。

写入接口必须携带 owner token。token 校验、目标进程身份校验、记录再读、写入、回读、回滚和结果确认都处于同一个临界区；旧页面在新页面接管后只能得到失效错误，不能触碰游戏进程。前端卸载页面时将精确 token 放入全局释放队列，失败后按 `250 ms` 到 `8 s` 指数退避重试，只有确认成功或后端判定已失效时才删除。应用关闭时再次强制恢复全部已知入口；恢复失败会阻止正常退出并显示诊断信息。

进程身份使用 `{PID, Created FILETIME}`，不能只依赖可能复用的 PID。远程保存超时、部分读写、代码洞冲突或恢复结果不确定时，该进程身份会被隔离；在游戏进程真正重启以前不再接受新的 Acquire 或写入。

### 4. 数据安全

- 祝福三个槽位只接受目录中已知 hash，空槽使用游戏的 EmptyHash 且等级必须为 0。
- 第一槽不可为空；等级分别受槽位上限与词条已确认上限约束。
- 新增词条使用 GBFRDataTools `ids.txt` 中的真实 `SKILL_*` 标识和 hash，不使用合成 ID。
- 因子数据修复只落入能被本地解包数据验证的映射；未知组合继续失败关闭，不用“可写”伪装。
- 读写进程内存必须检查实际传输字节数，部分读写视为失败。

## 前端设计

新页面复用现有羊皮纸设计系统，不引入第二套颜色或控件：

- 顶部共享连接状态条：进程、钩子、捕获记录三层状态；主要动作只有“开始捕获/停止捕获”。
- 中部一张记录卡，固定三行词条。每行并排显示当前值、目标选择器和等级；窄宽度自动纵向堆叠。
- 变更摘要默认折叠，只在存在差异时显示数量，避免永久占据空间。
- 底部粘性操作栏仅保留“放弃修改”和“写入当前祝福”；按钮等宽、键盘可达，错误与成功消息使用 `aria-live`。
- `canSubmit = 已连接 && 已捕获 && 指针未过期 && 有改动 && 三槽合法`。
- 写入成功后清空草稿、标记记录过期并把焦点移回“重新选择祝福”的提示。

响应式验证宽度为 375、768、1024、1440 像素；最低窗口高度下只能有一个主要滚动容器，不允许文字重叠、卡片横向溢出或操作按钮不可达。
