# NidasBot CT 0.8.4 运行时能力集成设计

日期：2026-07-19

## 目标

以本地 `GFR_v0.8.4_CHS_祝福石修改hotfix(Uno.ct` 为功能事实来源，独立重写其中能在 GBFRDataTools 内安全验证、完整恢复的能力；不执行或嵌入 Cheat Engine Lua/Auto Assembler，不复制 CT 的界面。新增能力继续使用项目现有的羊皮纸、金色、双栏桌面设计，并明确分入存档修改、实时内存和工具三类。

本次视用户此前“自行设计、直接完成、不逐项询问”为对推荐方案的预授权。产品原则是：已有能力只重分类和复用，新能力必须有真实后端、回读与恢复，无法证明安全的 CT 项不显示成可用按钮。

## 来源结论

- CT XML 自述脚本版本 `0.8.4`、游戏版本 `2.0.2`、日期 `2026-07-16`、Cheat Engine 7.5。
- 文件共有 370 个记录节点；去除主脚本和两个外链后有 149 个产品脚本、150 个直接字段。归一化后约 115 个叶子动作，其余是控制器、参数、标题和说明，并非 370 个独立功能。
- CT 的六个原始组全部操作游戏进程，没有 `SaveData.dat` 离线解析器。项目现有离线因子、祝福、配装、养成、记录、备份能力不应被 CT 风格页面替换。
- `0.8.1 → 0.8.4` 的关键新增包括：战斗辅助限制、实时自定义因子、任务内召唤限制、任务得分倍率、支线目标奖励、三角色蓄力调整、Steam 成就、选中素材/关键物品和祝福查看 hotfix。项目已以更安全方式覆盖因子生成和祝福查看；其余按风险分层。
- CT 0.8.4 已把玩家结构 `+0x9C` 更正为“共鸣点数/RP”。项目当前仍显示 CP，必须同步修正并兼容旧前端 ID。
- CT 内置 ID 表存在重复键和中文错配，不能直接导入：项目的本地 2.0.2 解包表、真实存档和游戏 EXE 继续拥有更高权威。

## 信息架构

顶层仍保持三类：

1. **存档修改（离线）**：物品与武器、因子、祝福、配装预设、角色记录、任务记录。给已有页增加“覆盖 CT 对应结果”的准确说明，不复制 CT 的运行时生成器。
2. **内存注入（实时）**：保留因子、祝福、配装录制、召唤石、上限突破、实时资源、怪物工具；新增三个页面：
   - `ctCombat`：战斗规则与通用动作，包括闪避、格挡、Link、召唤限制、部位破坏等。
   - `ctCharacters`：按真实角色分组的专属机制开关，组内折叠，支持搜索。
   - `ctQuest`：任务、掉落与便利功能，包括倒计时、宝箱、结算、支线奖励、因子合成等。
3. **工具与设置**：版本适配、EXE 维护、语言。新增 CT 覆盖摘要只作为版本审计信息，不提供无后端按钮。

三个新页共用 `CT084Features.vue`，由 `mode` 决定数据子集；页面不把 50 多张卡片一次铺开。桌面宽屏采用左侧分组/搜索与右侧当前分组列表，窄屏改为分段按钮和单列列表。每张卡只显示名称、角色/分组、状态、开关和必要参数；AOB/RVA/字节放入折叠技术详情。

## 可集成范围

### 第一层：声明式直接补丁

实装生成器确认 CT 有 64 个不依赖代码洞的 AOB 补丁候选；预审漏掉了带 `Async="1"` 属性的 CT 32556「MSP、境界点数不减」。排除以下四项后，目标是集成 **60 个新直接补丁**：

- 欧根“爆裂射击秒蓄力”：CT 明示关闭会崩溃。
- “无限重复任务”：CT 明示实验且可能有 bug。
- `100% 巴武掉落率`：项目已有 owner token、AOB、回读和恢复实现。
- `无限 Link time`：项目怪物页已有独立实现，避免同址冲突。

纳入的直接补丁覆盖战斗、任务、便利和大部分角色机制。巴萨拉卡、巴恩、希耶提之间的共享冲突必须写入冲突组；冲突项只能二选一，不能相互覆盖。

### 第二层：0.8.4 重点代码洞

在直接补丁注册表稳定后，独立实现并测试：

- 任务得分倍率；
- 人物动作速度（当前角色/全队）；
- 玩家/队伍只读监测（HP、最大 HP、奥义、闪避、坐标），只读不写；
- 当前选中素材/关键物品的受限数量编辑。

自动完美连招、三角色跳过蓄力、Steam 成就和解锁人物本轮不伪装成已完成：前两者需要更复杂的现场寄存器/角色状态实机互证；后两者不可逆且外部状态边界不清。它们进入最终差距报告，而不是可点击 UI。

## 通用补丁后端

新增独立文件，避免继续膨胀 `app.go`：

- `ct084_catalog.go`：嵌入并校验声明式目录。
- `ct084_pattern.go`：解析 CT 的半字节通配 AOB（每个 nibble 可为十六进制或 `x/?`），在模块内唯一匹配。
- `ct084_runtime.go`：进程租约、扫描、启用、禁用、状态和批量恢复。
- `data/ct084_patches.json`：只保存独立重写所需的功能名、CT ID、分组、AOB、补丁片段、冲突组与来源版本，不保存 Lua 或完整 CT。

每个补丁可以有多个 AOB 站点和多个相对片段。启用事务必须：

1. 持有全局 `liveMemoryWriteMu` 和当前页面 owner token；
2. 绑定 `{PID, Created FILETIME}`；
3. 对每个站点做唯一 AOB 命中并保存完整原字节；
4. 写前确认所有站点仍是原始状态；
5. 逐站写入并立即回读；
6. 任一失败时逆序恢复已写站点并回读；
7. 只有全部站点验证成功才标记启用。

禁用只允许恢复当前进程实例中本工具保存的原字节。地址、进程或现有字节任一不符都拒绝覆盖并隔离当前进程写入。页面释放、主动断开、进程替换和应用退出均恢复该 owner 的全部补丁；恢复结果不确定时保留租约并阻止静默退出。

扫描使用原始模式；启用后的状态读取依赖本工具持有的地址与原字节，不猜测未知原始通配字节。应用异常退出后由游戏重启恢复，不尝试从通配模式反推原指令。

## 参数化与只读能力

- 任务得分倍率限定 `0.1–20.0`，浮点必须有限；代码洞内倍率更新同样写后回读。
- 动作速度限定 `0.25–3.0`，默认只作用玩家；“全队”是显式独立选择。
- 队伍监测连续读取三次稳定快照才标记“已验证”，字段范围和 NaN/Inf 均失败关闭。
- 选中物品只允许项目目录中已知素材/关键物品 ID，数量为 `0–999999`，状态只能使用实机验证枚举；整条记录写入、保存、回读，成功后使捕获指针失效并要求重新选择。

## 前端交互

- 新实时页复用一个连接条和 owner token 生命周期。
- 首次启用本轮 CT 功能时只显示一次“仅离线/单机使用”确认；来源作者也明确要求 offline only。
- 开关等待后端回读后才改变视觉状态，禁止乐观伪成功。
- 搜索匹配中文名、角色名和分组；角色页默认只展开当前选择的角色组。
- 互斥功能在对方启用时显示具体冲突名称并禁用，不用笼统“不可用”。
- 错误与成功写入 `aria-live`；按钮有键盘焦点、禁用态和忙碌态；不使用 emoji 充当图标。
- 响应式门禁为 375、768、1024、1440 像素；页面只有一个主滚动容器，标题、状态和操作区不重叠。

## 资产与视觉

三个新页使用独立、功能匹配的立绘和 Q 版贴纸，不复用现有页面映射：战斗页表现“校准战斗规则”，角色页表现“检查队伍机制”，任务页表现“规划任务与宝箱”。具体人物、动作、道具和警示语由配套资产设计规格决定。

## 验证标准

- 目录测试固定 60 个新直接补丁、唯一 CT ID、合法 nibble AOB、站点/片段边界、生产目录 SHA-256 和冲突组。
- 纯函数测试覆盖 nibble 通配、跨块扫描、零命中、多命中、补丁片段合并和越界拒绝。
- 事务测试覆盖多站点启用、第二站失败回滚、回读不一致、禁用时外部改写、进程身份变化和全量退出恢复。
- 本地 `granblue_fantasy_relink.exe` 上离线扫描全部注册 AOB，记录唯一命中率；不匹配项不进入生产目录。
- Go、Node、Vite、Wails 全套测试和构建通过；运行游戏可用时再做只读扫描和显式开关/恢复实测，不能运行时明确报告环境边界。
