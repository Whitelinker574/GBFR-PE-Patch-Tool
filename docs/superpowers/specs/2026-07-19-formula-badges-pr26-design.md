# 配装数值、合规报告与称号记录集成设计

## 目标

本轮解决三个可独立验证的问题：修正解包格式串中的数值倍率、把现有严格校验器的结果显式呈现在配装编辑页、从 PR #26 提炼离线称号记录编辑能力。新版配装模拟器仅作为交叉参考；本地 DLC 2.0.2 解包表、真实存档结构和现有写盘事务仍是数值与写入的主证据。

## 数值计算

`trait_values.json` 中的格式串不只有小数格式，例如昏厥使用 `昏厥值+{0:10}`。这里的 `10` 是显示/面板倍率，原始 Lv45 值 `10.0` 应进入人物面板为 `+100`。解析器会把格式说明解析为结构化倍率，并同时用于效果文本与 `BonusComponent`，避免“文字正确但总计错误”或反向不一致。

不会复制外站的六类固定角色/武器基础值、分段插值或未给出来源的舍入顺序。现有 410 武器的等级、突破、觉醒、超凡和技能表继续作为唯一计算路径；界面只补充技能的当前有效等级与解锁来源说明。

## 合规检测

不新增一套宽松规则。配装页把现有后端预检结果汇总为逐槽报告：每个因子槽显示主词条、副词条、等级和 `合法 / 固定组合 / 未证明 / 不可写` 状态及原因；导入和写入共用同一验证入口，未知目录继续 fail-closed。报告明确区分“可写”与“自然掉落合规”，避免把修改器上限误称为游戏合法上限。

## 称号记录

PR #26 不直接合并。仅提炼三项事实：称号解锁向量 `5801`、奖励领取向量 `5814`、已查看向量 `5816`，以及 1616 条中英目录。新功能作为现有“任务记录”页面内的第二个标签，不增加顶层页面、重复立绘或深色旧组件。

后端流程：

1. 完全退出游戏的后端门禁；
2. 在单一称号写事务中读取文件并要求三个向量均为 `UnitID=0`、长度 1700、无重复；
3. 只修改请求的 `5801` 位，并默认同步相同 `5816` 位；`5814` 字节级保持不变；
4. 修复校验和，复用 `SaveData.Write` 的整组快照、同档备份和原子替换；
5. 回读请求意图、已查看状态、奖励向量和向量边界；失败时不无条件覆盖可能由后来写入者产生的新文件。

目录以固定条数、ID 范围、唯一性、名称完整性和源文件 SHA-256 测试锁定。目录存在名称不等同于已完成游戏内领取验证；UI 会把“解锁记录”和“奖励领取记录”明确分开。

## UI 与响应式

继续使用当前羊皮纸 token、语义化按钮、可见焦点和统一确认弹窗。称号列表采用固定行高窗口化渲染；筛选、语言和状态变化会同步重置真实滚动容器。宽屏使用紧凑工具条与列表，窄屏把筛选与批量操作折行，不挤压右侧固定立绘。遵循 375、768、1024、1440 像素四档检查。

## 验收边界

- 昏厥 Lv1 与 Lv45 的文本、结构化分量和最终面板一致。
- 武器技能仍来自本地表，外站冲突值不会进入代码。
- 合规报告与写入预检对同一草稿给出一致结论。
- 称号目录契约、畸形存档拒绝、单项/批量、已查看同步、奖励不变、并发与真实存档副本测试通过。
- 前端全量、Go 全量、Wails 原生多尺寸矩阵和正式构建全部新鲜通过。
- 报告诚实保留真实游戏进程 E2E、未证明公式和缺失精确图标的边界。
