校准日期：2026-07-21  
游戏版本：DLC 2.0.2

现场分支：`field/runtime-formula-calibration`

## 结果摘要

首要故障已经定位并修复。`4D0A60C3` 不是错误的目录 hash；它与伊欧现场运行时状态 map key/runtime ID 精确一致。连接失败的实际原因有两层：

1. 后端旧定位器把 `status+0x59F0` 当成必须等于目录 hash 的身份字段，并以 ready/eligibility 作为硬过滤条件。现场伊欧对象的该候选 hash 为 `00000000`，所以真实状态对象被错误淘汰。
2. 后端修复后，旧版前端仍可能显示上一采样会话的值。单一 `observing` 布尔值跨会话复用，旧请求会抑制或覆盖新 session 的响应。现使用 session token 与递增 epoch 隔离所有异步观察，并在连接、关闭和卸载时使旧请求失效。
3. 配装估算原先只信任磁盘静态成长。现场德吉塔存档掩码只解析出命运篇章 `HP +10 / 攻击 +3`，而游戏已加载对象的真实聚合是 `HP +640 / 攻击 +165`，造成最终 `HP -757 / 攻击 -309` 的系统性误差。现从每个角色状态对象读取基础值、命运篇章、Master 固定成长和角色强化聚合，连续三次一致后才覆盖离线值。
4. 武器表等级也不总等于角色实际生效等级。伊欧的重制终末武器静态浩劫为 Lv30，运行时有效等级是 Lv32（攻击 `+64%`、上限 `+450%`、HP 阈值 `170000`）；德吉塔同类武器则保持静态等级。程序现按角色读取有效武器技能和武炼结晶快照，不再把一个角色的修正套给所有角色。

新版定位器先受 2.0.2 EXE 四段字节守卫约束，再安全枚举 manager ID 向量和完整 map。精确 map key 是主身份；`+0x59F0`、ready 和 eligibility 保留为诊断证据，不再错误否决精确 map-key 对象。容器损坏、地址溢出、版本不符、字段越界和三次读取不稳定仍全部失败关闭。

## 两角色现场逐项对照

| 角色 | 目录 hash | runtime ID/map key | HP | 攻击力 | 暴击率 | 昏厥值 |
|---|---|---|---:|---:|---:|---:|
| 伊欧 | `4D0A60C3` | `4D0A60C3` | `132977` | `111259` | `124` | `306` |
| 姬塔 | `A4ACBA76` | `A4ACBA76` | `102131` | `25887` | `115` | `216` |

配装页面用同一组角色独立运行时输入重算后也逐项一致：伊欧保存槽 06 为 `132977 / 111259 / 124 / 306`，姬塔当前队伍为 `102131 / 25887 / 115 / 216`。普通 HP/攻击面板在 float32 乘区后四舍五入；伊欧重制终末武器先把普通攻击小计四舍五入，再应用独立末乘区并向零截断。该两层量化同时满足两个角色，未加入角色专用补丁常量。

伊欧另一武器样本为 `134588 / 71468 / 123 / 329`，恢复原配装后返回原值。伊欧目标实验配装为 `159573 / 111259 / 108 / 306`。一个全新只读进程句柄能立即取得目标实验配装，而旧工具界面仍显示上个 session，直接把问题范围收敛到前端异步所有权而非 manager/字段布局。

## 专精现场实验

实验只改游戏配装预设中的一个节点，随后使用游戏自身“读取该配装”应用；每组已验证结果都执行 A1/B1/A2/B2。实验完成后关闭游戏、恢复整组原始存档并逐文件校验。

### `1F52146F`：伊欧“昏厥值 +0.4”

| 阶段 | HP | 攻击 | 暴击 | 昏厥 raw | 显示昏厥 |
|---|---:|---:|---:|---:|---:|
| A1（有节点） | 159573 | 111259 | 108 | 30.599998 | 306 |
| B1（无节点） | 159573 | 111259 | 108 | 30.2 | 302 |
| A2（有节点） | 159573 | 111259 | 108 | 30.599998 | 306 |
| B2（无节点） | 159573 | 111259 | 108 | 30.2 | 302 |

结论：原表 float32 动作值 `0.4` 在状态对象保持 raw 尺度，人物面板显示 `×10`，所以最终显示贡献为 `+4`。禁止把特定屏幕值或绝对地址硬编码。

### `A66398F9`：防御专精“最大 HP +15000”

| 阶段 | HP | 攻击 | 暴击 | 昏厥 |
|---|---:|---:|---:|---:|
| A1/A2（有节点） | 159573 | 111259 | 108 | 306 |
| B1/B2（无节点） | 140673 | 111259 | 108 | 306 |

结论：当前配装的最终差值为 `18900 = 15000 × 1.26`。这证明平坦 HP 节点进入当前 HP 百分比层；`18900` 只属于这个配装样本，程序不能把它当成通用常量。

### `E0E6FF0C`：专注范围内减伤

有/无该节点时，城镇人物面板四项均保持 `159573 / 111259 / 108 / 306`；同一状态对象前 `0x6000` 字节的受限扫描也没有直接出现节点 hash。

证据等级只能写作“城镇面板负观察”。尚缺训练场中专注范围未激活/已激活的成对受击样本，包括受击前 HP、受击后 HP、实际伤害和当前 Buff。四项面板不变不能推断技能无效。

## 专精解锁边界

`10/10/10/20 = 50` 是四组槽数组的结构上限，不是当前解锁等级。真实存档联表已验证：

| 总 MSP | Master Lv | 当前容量 |
|---:|---:|---|
| 0 | 1 | `1/0/0/0` |
| 136500 | 20 | `10/10/0/0` |
| 3309499 | 50 | `10/10/10/20` |

现场伊欧 50 节点与 49 节点预设分别显示 `10/10/10/20` 和 `10/10/10/19`。仍未闭环的是：仅凭总 MSP 为 0 的离线字段，无法区分“系统尚未开放”和“已开放但未获得 MSP”；需要额外进度标志或成对 UI 样本。

另外确认：直接改活动队伍记录中的 `IDType 3007` 不会即时刷新游戏当前专精。可靠、可逆的现场路径是修改一个配装预设，再由游戏 UI 读取该预设。编辑器后续写入逻辑不得把队伍记录误当成当前运行时状态。

## 改动

- 新增结构化运行时对象枚举与脱敏目录：ID 向量、map key、ready、eligibility、候选 hash、目录名/hash/runtime ID、四字段类型/相对偏移/稳定性。
- 定位器改用精确 map-key 身份，候选 hash 与状态 flag 不再错误硬过滤。
- 自动变化记录支持稳定前值、过渡期、稳定后值和重复性分析；保留严格 A1/B1/A2/B2。
- 24 KiB 差分限定在同一状态对象，并只导出相对偏移、类型、位模式和变化，不导出原始内存。
- 版本化布局绑定游戏 EXE SHA-256、四处代码守卫、访问链、字段类型/偏移/尺度、样本角色数和证据等级。
- 修复前端采样 session 竞态，旧观察不能覆盖或抑制替换会话。
- 配装页新增角色独立运行时成长、有效武器技能、武炼结晶审计，以及草稿/实读/差值四项对照。
- EX `1F52146F` 同时显示解包原始 `0.4` 和已验证面板 `+4`（`f32 ×10`），避免把原始单位误当显示单位。
- 修正普通面板四舍五入与重制终末武器独立末乘区截断的两阶段量化。
