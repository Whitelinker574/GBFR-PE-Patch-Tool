## 结论先行

本轮要求中的应用层、离线存档层、配装编辑器、界面统一、数据校验和运行时安全改造已经落地；相对 BitterG v1.8.4 的“新祝福生成崩溃修复”，本项目已实现同等行为并增加了自动停 Hook、进程归属、整条事务写入、回读、回滚与不确定状态隔离。

当前结论是：

- **离线存档与应用 UI：可交付。** 因子、祝福、配装、专精、召唤石、导入导出和真实存档副本写入均有自动回归。
- **运行时代码：静态与模拟门禁通过，但仍是“有条件可发布”。** 当前机器没有安装/运行游戏 2.0.2，无法完成最后一次真实进程内注入、铁匠铺切页和退出恢复验收。
- **人物最终 HP/攻击：仍是显式估算，不是伪装成精确公式。** 基础值、武器四段成长、最终 `float32` 向零截断等路径已经闭环；百分比 producer 的乘区及中间量化顺序尚未完全闭环。
- **并非所有图标和效果都有官方资源。** 项目现在只使用能精确联表的官方图，不再拿相似图冒充；缺失项会回退或显示未计入警告。

因此，“已知代码缺陷与本轮功能缺项”已经收束，但不能把“缺少游戏本体的真实进程验收”和“尚未逆完的游戏公式/资源”写成已经完成。

## 本轮完成的主要工作

### 1. 配装编辑器重做

- 编辑配装改为进入独立大页面，正常配装列表仍保留右侧角色立绘；编辑页隐藏立绘，把空间留给配置和结果。
- 默认进入“构造因子”模式，不再默认从背包选择。
- 上方固定展示 12 个槽位、当前因子以及主/副词条；下方才是搜索、构造或背包候选，符合“先看配置，再替换”的操作顺序。
- 背包候选改为独立卡片布局，避免因子堆叠；构造模式以替换槽为核心，写入时才生成新因子并绑定到配装。
- 支持真实武器、四个可装备主动技能槽、完整角色技能候选、武器技能、四个召唤石、12 个因子和专精配置。
- 单个角色的一套配装支持 JSON 导入/导出；保留槽位顺序并报告缺失实例，不做用户已明确取消的批量导出。
- 写入采用计划、备份、写入、校验和回滚流程；构造出的新因子、技能、专精、召唤石与角色记录同步写入。

### 2. 专精规则与界面

- 专精放在因子配置下方，可完整展开，结果继续汇总到右侧。
- 展示三条专精方向及 R1、R2、R3、EX 阶段，不再只显示两个方向或缩略结果。
- R1 的三方向根技能可以并存；从 R2 起只能形成一个主方向，R3 必须沿用该方向。
- 没有选择主方向时，R2/R3 的普通子节点允许保留，但不会伪造对应方向的根效果。
- 等级配额、阶段依赖、方向冲突和导入数据都由前后端共同校验。

### 3. 属性结算与来源说明

- 头像下方显示 HP、攻击力、暴击率、昏厥值、伤害上限五项结果。
- 因子、武器技能、专精、角色上限突破、召唤石均进入汇总；武器技能不再遗漏。
- 同类加成先聚合，再按等级曲线封顶；同时显示投入等级、有效等级和溢出等级。
- 伤害上限保留普通、能力、奥义三类拆分，不把不同上限强行合成一个虚假值。
- 右侧提供来源账本，明确当前默认计算范围；未知召唤效果会显示哈希及“未计入”，不会静默消失。
- 公式证据、RVA、字段、舍入点和真实回归值写入 [`FORMULAS_2.0.2.md`](FORMULAS_2.0.2.md)。

### 4. 因子、祝福与数据真值

- 修正 `GEEN_156_24`：本地 2.0.2 `gem.tbl` 证明其主词条为 `SKILL_023_00 / 0xCAC6AFF2`，不再误用 `SKILL_156_00`。
- 角色专属因子 91 项全部按本地 `gem.tbl` 与二词条 lot 联表：25 项固定副词条，66 项使用已验证的随机池；固定副词条不能省略或替换为任意词条。
- 无法证明自然生成合法性的条目不能进入构造队列；等级限制只公开自然范围。
- `0x40223C28` 按本地 `ids.txt` 和生成表修正为 `SKILL_143_00`，不再错误标成 `SKILL_166_50`。
- 因子/祝福离线生成器增加实例级串行化、队列副本返回、数量后端上限、写后回读和失败恢复；失败重试不会重复追加。

### 5. 运行时写入安全

- 因子、祝福、召唤石、上限突破、货币、药水、素材消耗、巴武掉落和怪物增强写入统一加入进程租约/所有权校验。
- 写入前后复核进程 PID、模块基址、捕获地址和旧值；页面令牌不能跨进程复用。
- 同一进程上的互斥运行时功能统一进入冲突门禁，避免两个页面同时改相同代码或状态。
- 所有关键写入增加回读；失败时尝试恢复原值并再次回读。无法证明回滚成功时会隔离当前进程实例，拒绝后续写入。
- 断开、页面停止、应用退出和进程重启走同一生命周期；恢复失败不丢弃令牌，可继续重试。
- 上限突破代码洞保存/恢复 `RFLAGS`、`R10`、`R11` 并保持栈平衡；每个槽位按 16 字节整块写入和回读，部分失败会回滚已写槽位。
- 伤害统计中的本进程裸指针访问改为系统内存读写 API，消除 `go vet` 的不安全指针告警。

### 6. UI 统一与自适应

- 18 个页面统一到同一套纸张/金色设计令牌、按钮、输入框、卡片、状态和弹窗原子组件。
- 调整字体、字号、行高、数据字体、文本换行、面板宽度、间距和密度；减少过长、过宽、只占空间的说明框。
- 窗口在 960×620 至 1680×1050 间按侧栏、内容列和卡片断点自适应，避免固定宽度导致文字重叠或功能被裁掉。
- 页面截图来自实际 Wails WebView2 的 CDP `Page.captureScreenshot`，不是浏览器仿页面或桌面区域猜测截图。
- 英文设置页不再承诺“完整翻译”；当前英文明确标成部分覆盖。

## 还有没有没做完的

有，以下是不能用代码单方面伪造“完成”的边界，按优先级排列。

### P1：需要游戏环境才能关闭

1. **2.0.2 真实进程验收**：需要在目标 EXE 指纹对应的游戏中依次测试因子、祝福、召唤石、上限突破、货币/药水、素材/掉落、怪物增强的启用、写入、切页、停止、断开和退出恢复。
2. **新祝福铁匠铺崩溃场景复验**：需要实际执行“启用 → 选中 → 写入自动停用 → 切换武器强化/其他铁匠铺页”，确认与 v1.8.4 的真实崩溃场景完全闭环。

### P1：公式逆向仍未闭环

1. HP/攻击百分比效果在上游如何变成绝对贡献、不同乘区顺序及每次中间 `float32` 量化点尚未全部定位。
2. 2.0.2 的 ATK↑/DEF↓ 额外伤害、echo、supplemental 与各动作上限的最终顺序尚未闭环。
3. 当前页面会明确标记估算和未计入项，因此这是精度边界，不是静默给错值。

### P2：给定数据/资源本身的缺口

1. 1 个可玩主动技能、2 个词条、4 个武器、11 个物品缺精确官方图标。
2. 60/230 个召唤主加护没有足够的本地权威效果定义；目前已逐项警告但未猜值。
3. 因子目录有 187 项，而本地 2.0.2 `gem.tbl` 可直接联表 184 项；另外 3 项不能仅凭名字推断自然生成规则。
4. `secondary-trait-rules.json` 仍保留少量低/中置信历史规则和 TODO；本轮已把 91 个角色专属因子改为本地表的 25 个固定池与 66 个随机池，其他无法证明的组合继续 fail-closed，不能宣称整个游戏的所有随机池都已穷尽。

### P2：兼容实验室

任务结算倒计时/零帧开箱、无限挑战、脸部/其他皮肤紫色符文、全称号、全队伤害统计、任务得分倍率、强制支线奖励和任务内倍率保留了旧资料或实验实现，但未完成 2.0.2 生产验收，UI 已明确隔离为兼容实验室。

### P3：非阻断工程项

1. 英文界面仍是部分覆盖。
2. 1269 个背包因子使用完整卡片列表；功能正确，但还可做虚拟滚动和大数据性能基线。
3. 前端主包仍有大 chunk 警告，可继续拆分异步页面和资源索引。
4. `go.mod` 使用 Wails 2.12.0，而本机构建 CLI 为 2.13.0；当前能构建，但发布环境应统一版本。
5. Windows race detector 需要安装 C 工具链并启用 CGO 后补跑。
6. 怪物增强 Hook 停止时会恢复并验证入口字节，但出于避免 DLL 卸载竞态的考虑，远程代码洞及已加载模块暂不在游戏存活期间执行 `VirtualFreeEx`/卸载；单次启用会留下很小的远程分配，游戏退出后由系统回收。
