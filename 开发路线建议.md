## 一、项目现状与定位

**健康度好**：149★、9 fork、每天发版、只有 **3 个 open issue**。本地 v1.9.1 领先公开 v1.8.0。

**生态定位（调研结论）**：BitterG 是整个 GBFR 生态里**唯一横跨「离线存档编辑 + 运行时内存 + 怪物增强 + DPS 统计」四类的整合式开源 GUI**。其余项目全是单点工具：

| 类别 | 代表 | 状态 |
|---|---|---|
| 数据地基 | **Nenkai/GBFRDataTools**（C#, MIT, 139★, v2.0.0 已跟进 DLC）+ nenkai.github.io/relink-modding 文档站 | 权威 sigil/trait/DLC hash 的唯一正源 |
| 内存训练器 | NidasBot CT（fearless，需 CE，最勤） | 战斗/角色作弊最全 |
| DPS 统计 | false-spring/gbfr-logs（Rust+Tauri，架构同源）、nyaoouo/GBFR-ACT（328★） | 可参考内存偏移 |
| 存档编辑 | limitcool/gbfr-tool（Svelte，36★）、XzireAstra（闭源疑付费） | 功能面窄 |

**独占护城河**：PE 补丁 + 怪物增强 + 离线存档编辑 + DPS Meter —— 这几项生态里几乎没有对手。
**短板**：每个单项都不如该领域专精工具（DPS 不如 gbfr-logs、内存作弊不如 CT）；数据权威性依赖 Nenkai。

## 三、贯穿性技术债：名表数据未从权威源干净重导（最该先解决）

多个用户抱怨（#17 躲避性能重复、#6 天星之练/炼）、以及数据审计发现的大量问题，**根因是同一个**：`sigil_ctdata.go` 等名表不是从 GBFRDataTools 权威数据干净生成的，导致：
- **系统性重名**：两 hash → 同名（躲避性能×2、剑圣的练气×2、永暗藏匿者×6、Life on the Line "+" 变体×3…）。
- **DLC 缺失**：天星系列等 DLC 因子/特性名表里根本没有（用户报的"天星之炼"无法修，因为条目不存在）。
- **零散错标/错译**：已修 2 条（守护V+/坚持V+），但"+"变体块系统性错标未根治。

**根治方案**：用 `Nenkai/GBFRDataTools` 解包 `data.i` → `system/table/*.tbl` 转 SQLite，重新导出全量 sigil/trait/summon/weapon 的 hash↔名映射（含 DLC 2.0），一次性替换现有手工名表。这能同时消除：重名投诉、DLC 缺失、翻译错标、以及 D-1 未根治的"+"变体块。**投入产出比最高的一次性工程。**

## 四、新功能方向（按 难度×实用性×复用度 排序）

### 第一优先级 —— 契合已有能力、社区呼声最高

1. **配装模板库 + 一键毕业配装（扩展 #22）** ★★★★★
   - 社区最大空白：所有工具都是"手动填因子 ID"，没有一个做"选角色→一键套用当前版本 Meta 毕业配装"。
   - 你已有因子生成 + v1.8.0 的配装导出/导入地基，只需：(a) 内置各角色 Meta build 预设库；(b) #22 想要的"分享码"（把配装编码成短码，别人粘贴即生成）——比文件式更易传播。
   - **联动**：v1.8.0 release note 预告的"读 DPS 软件 sqlite 抄队友配装"，那个软件就是 **gbfr-logs**（Rust+Tauri 开源，有 SQLite 日志），可解析其库拿队友实时配装。

2. **仓库实时编辑补齐：当前选中武器（CT NBGFR079）+ 无限物品（NBGFR077）** ★★★★★
   - 与工具已跑通的"因子/召唤石实时 hook"是**同一套范式**（AOB 捕获选中指针→改字段→调保存函数），复用度最高、成本最低。

### 第二优先级 —— CT 有稳定 AOB、玩家想要

3. **战斗核心组**（无敌/锁血/一击必杀/伤害倍数 CT NBGFR003、无限buff/免疫debuff NBGFR005、无冷却 NBGFR004）★★★★
4. **任务组**（一键发现宝箱/自动收集/跳过结算/自动完成支线/无限挑战时间 CT NBGFR082–087）★★★★ —— 刷本刚需
5. **无限/100% Link time**（NBGFR011/012，区别于怪物增强里的 link_time）★★★★

### 第三优先级 —— 低成本补全

6. 体验类：MSP/CP 不减（092/094）、因子合成强制满级（093）、突破专精上限（096）、无视钥匙开箱（089）、动作变速（090）★★★

### 长尾 —— 按需

7. 角色专属作弊（CT NBGFR019–076，约 30 角色）★★ —— 数量巨大、受众窄，按热门角色分批做

## 五、旧功能现状：核心实时编辑正常，只有 6 个"待适配"功能需按版本重校地址

> **重要更正（据仓库作者反馈 + 复核公开仓库源码）**：先前把"硬编码 RVA"一律说成"脆弱/失效/该改 AOB"是**错误框架**。实情是——
>
> **本工具是版本锁定设计**：RVA 针对每个游戏 build 验证过（`sigil_memory.go` 注释 `Current game build verified at granblue_fantasy_relink.exe+345157`），版本号 `v1.9.1-local-dlc202-ct081` 就编码了对应的 DLC/CT 版本。配合每天发版的高频节奏，**每次 DLC 更新后重新校准 RVA 再发版**，是完全成立且正常工作的策略。**因子/召唤石/货币/上限突破等核心实时编辑当前工作正常，不是待修 bug。**

### 5.1 当前正常工作（无需动）

- **因子实时编辑**（`sigil_memory.go`，硬编码 RVA `0x345157`，已按当前 build 验证）✅
- **召唤石实时编辑**（`summon_memory.go`）✅、**货币**（AOB，=CT NBGFR001）✅、**上限突破**（AOB 双特征，=NBGFR091）✅、**巴武掉落**（AOB，=NBGFR088）✅、**材料不消耗**（RVA+AOB 回退）✅
- 说明：`sigil_memory.go` 里那个 `scanSigilMemoryPattern` + `sigilMemorySelectedPattern` 是**没接进主路径的 fallback 死代码**，且其数组本身有长度不一致（26 vs 27）。因为主路径走已验证的硬编码 RVA，**它坏不坏不影响功能**——属可删的死代码，不是待修 bug（详见 `代码审查报告.md` C-4）。

### 5.2 真正"待适配"的 6 个功能（作者已在 UI 用"兼容性实验室"模式圈出）

`MiscTools.vue` 的 `showOutdatedFeatures`（`mode === 'compatibility'`）把这些标为"等待适配"，默认仅建议扫描诊断：
- 团队伤害记录、任务结算倒计时/零帧开箱、无限挑战、脸部符文显示、游戏内全称号、其他皮肤紫色符文。
- **这些才是"DLC 更新后暂未重新校准地址"的功能**。修法 = 按作者常规流程，为当前 build 重新验证/更新它们的 RVA/AOB（部分用硬编码如无限挑战 `0x278A6DE`、紫符文 `0x9175B6`，需重定位）。这是**每版维护的常规活**，不是架构问题。

### 5.3 是否值得改用 AOB？——可选优化，不是必须

- 版本锁定的硬编码 RVA + 高频发版已经是**能用且作者惯用**的策略。改 AOB 是一个"降低每版重校工作量"的**可选优化**，不是修复"坏了的东西"。
- 若要做，优先级最高的候选是**怪物增强整组**（`dllmain.cpp` 全硬编码、无 AOB 回退，每版全靠手工重校）和**保存函数 `0x79D820`**（因子+召唤石共用的单点，本次已加 `validateRemoteFunctionStart` 启发式守卫兜底）。把这两处的 `expected`/`Original` 字节序列打通配做成 AOB，能显著减少每版维护量。但这是锦上添花，非燃眉。

## 七、建议的执行顺序（我的推荐）

1. **名表从 GBFRDataTools 重导**（第三节）—— 一次解决重名/DLC缺失/错标一大类 issue，是地基。核心实时编辑本就正常，这才是最该先做的一次性工程。
2. **配装分享码 + 毕业模板库**（#22 扩展）—— 社区最想要，契合已有能力。
3. **仓库实时编辑补齐**（武器 CT NBGFR079 + 无限物品 077）—— 复用现有已跑通的实时 hook 框架。
4. **重新校准 6 个"待适配"功能的地址**（5.2）—— 按每版常规维护流程，让兼容性实验室里的功能重新可用。
5. 战斗核心组（第四节第二优先级）→ 逐步补全。
6.（可选）怪物增强整组 + 保存函数改 AOB（5.3）—— 降低每版重校工作量，非燃眉。
