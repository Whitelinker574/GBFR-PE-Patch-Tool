# 全解锁存档 IDType 盘点：被工具忽略的可编辑系统

> 由 25-agent 解码工作流产出：解码全解锁存档 SaveData1.dat 的 609 种 IDType，
> 交叉比对 302 张游戏表 + 双存档差分，10 簇并行解码 → 逐项对抗验证（7 确证 / 7 否掉）。
> **原盘点结论止于文件级往返验证；当前分支虽已落地称号记录与角色永久强化只读链，但“游戏是否认账”仍未完成真实进程 E2E。**

## 汇总
原盘点通过全部解码系统 + 对抗验证找出 7 个「玩家可见、可编辑、当时工具未覆盖」系统。当前分支已把原 #2 称号记录安全落地，因此不能再写成“7 个仍未覆盖”：现状是 **1 个已落地、6 个仍为候选**。称号实现没有直接合并 PR #26，只锁定提炼 `5801/5814/5816` 三向量事实与 1616 条中英目录，并复用本项目写盘事务。

「改名(4413/834)」仍要双缓冲写入并处理 Binary1 完整性，是唯一 high effort 项。多个看似诱人的「收藏/图鉴」已被验证否掉：1452/1453 只是召唤石按类型登记层（全置 1 不给实物，真库存在 1456-1460）；2550/2551 与 2580/2581 实为任务清关记录，与工具已覆盖的 2570/2571 同族；4201/4202、5901/5902、6001/6002 分别是内部哈希字典、召唤子层、教程「已看过」标志。存疑但潜在高价值的召唤石存档库存与珍玩仍建议实机验证后再定。除称号已经复用的安全写盘路径外，其他候选落地时仍需重算 SlotData 尾部 10 段 xxhash64；改名额外触及工具模型之外的 Binary1。称号的文件回读也不能替代游戏实际认可，真实进程 E2E 仍待完成。

## 候选系统与当前状态（沿用原价值排序）

### #1 玩家显示名 / 改名
- **IDType**: 4413 (+834)　**实现难度**: high
- **是什么**: 玩家在联机资料卡 / 选档界面显示的自设昵称。存档里有两份副本(SlotData 4413 + Binary1 834)且恒相等。
- **编辑价值**: 玩家最想要的改名功能——修改自己的联机显示名。
- **风险/注意**: 中。必须同时改 4413(≤40 UTF-8 字节 +NUL)与 Binary1/834(≤63 字节 +NUL)，只改其一大概率不生效或不一致(Binary1 那份很可能才是选档/资料卡的读取源)。Binary1 完全在工具现有写入模型之外(不扫描、不打补丁、完整性机制未知)，需新写入逻辑并先确认 Binary1 是否另有校验，否则可能崩档。名字要按游戏字符上限截断、勿截多字节字符中间。游戏是否认此改动未实机验证。

### #2 已落地：称号 / 徽章（バッジ）记录
- **IDType**: 5801 / 5814 / 5816　**实现难度**: low
- **是什么**: 游戏「记录 / ノート」菜单里的称号 / 徽章图鉴。当前目录锁定 1616 条中英名称；存档用长度 1700 的 `5801` 解锁、`5814` 奖励领取、`5816` 已查看三向量保存状态。
- **当前实现**: PR #26 没有直接合并；只从锁定提交 `14ad0df` 提炼三向量事实与目录，目录再以条数、ID 范围、双语名称完整性和 SHA-256 `2aab04739254bebb26d48db9ec5f9cd804cda6a37a09654de2344cd41f4d833f` 固定。UI 作为既有“任务与称号记录”顶层页内的第二个标签，提供搜索、语言/状态筛选、单项与目录内批量操作，不增加顶层页。
- **写入边界**: 后端要求游戏完全退出，并要求三个向量都恰有一个 `UnitID=0`、长度 1700、每项只能为 0/1。事务只修改请求的 `5801` 位，并可同步同 ID 的 `5816` 已查看位；`5814` 奖励领取向量必须字节级保持不变。写入串行化，修复校验和，复用现有备份与原子替换，并回读验证请求意图和三向量边界。
- **仍未证明**: 目录契约、畸形输入、单项/批量、并发与隔离真实存档副本的测试已经写入仓库，但本报告不虚构尚未由主线回填的最新全量测试结果；文件级写回成功也不等于游戏进程已经认可，仍需实机进入称号页验证显示、持久化与奖励交互。

### #3 角色等级 / 经验一键满级
- **IDType**: 1303 / 1308 (勿单改 1309/1310)　**实现难度**: low
- **是什么**: 角色 / 队伍界面处处可见的角色等级与养成经验。
- **编辑价值**: 一键满级：跳过角色练级，统一写 8400000 / 100 即满级(满级时与经验曲线无关，很干净)。
- **当前只读覆盖**: 配装结算已按同一角色读取 `1308` 等级、`1309` 基础 HP、`1310` 基础攻击、`1312` `float32` 基础昏厥、`1313` 基础暴击，并继续联 `1318` Fate 与 `1323` Master 得到永久固定基准。这是只读结算链，不等于“一键满级编辑器”已经实现。
- **风险/注意**: 中。若以后实现写入，`1303` 与 `1308` 必须成对处理，不能只改其一指望补齐；更不能把配装只读链中已明确语义的 `1309/1310/1312/1313` 当成通用派生缓存随意覆盖。应仅对已获得角色（1305 含 `0x4000` 位）操作；游戏对强制满级、各基础字段重算和空槽的实际行为仍未实机验证。

### #4 人物档案图鉴全解锁
- **IDType**: 3801 / 3802　**实现难度**: low
- **是什么**: 游戏图鉴5大类中的「人物档案」页(姓名/声优/身高/种族/爱好/弱点/剧情笔记)，玩家在图鉴界面可见。
- **编辑价值**: 人物图鉴全解锁，用于100%完成度(连全解锁档也只36/39)。
- **风险/注意**: 中低。必须对全部39个unit同时写 3801=11 且 3802=true，只翻bool会留下「计数<11 但完成=true」的矛盾态，游戏内可能显示错误或被重算覆盖。数据在SlotData，写回重算hash(现有能力)。上线前建议实机改一角色(3801=11+3802=true)验证图鉴是否点亮。

### #5 道具交换所(妙具屋 / トレジャートレード)限购重置
- **IDType**: 4001 / 4002 (/4003)　**实现难度**: medium
- **是什么**: 交换所里显示「已售罄 / 剩余N件」的限购状态，玩家在商店界面可见。
- **编辑价值**: 把目标项的 4002 抬回 MaxStock 即可重置限购、再次购买稀有一次性奖励(武器 / 技能珠等)。
- **风险/注意**: 低(仅改已存在项)。关键联动：同一商品在不同存档落在不同UnitID(551项中413项换了槽)，必须按 4001==目标商品hash 定位到当前所在桶再改其4002，不能假设固定槽号；只覆写4002这4字节+FixChecksums，切勿往哨兵桶写新键新增商品(=需复刻开放寻址插入逻辑，易冲突崩档)。稀有一次性奖励改4002后能否真被游戏认可再购未100%证实(可能另有独立已领标志把关)，对消耗/素材类高度可信。

### #6 角色自定义通讯留言
- **IDType**: 1326　**实现难度**: medium
- **是什么**: 每角色一段玩家自填文本，极可能在联机中作为角色卡/自我介绍展示给他人(退一步至少是本地每角色文本标签；游戏侧有 communication 子系统佐证)。
- **编辑价值**: 查看 / 批量设置 / 清空角色留言。
- **风险/注意**: 低。每角色独立、不被其他IDType引用、仅存于SlotData单一写入点、无跨字段耦合。写入须保持241字节缓冲长度不变：新文本按rune边界截断到≤240 UTF-8字节(绝不截多字节字符中间)、其余NUL补零，再FixChecksums；UnitID→角色用IDType 1301映射。当前工具app层根本不读任何ByteTable，需新增字节缓冲编辑面。未实机目视确认显示位置。

### #7 多人通讯贴纸(Communication Stamp)收集
- **IDType**: 6301 / 6302　**实现难度**: medium
- **是什么**: 多人通讯用的表情贴纸，玩家在贴纸盘界面可见并可自定义排序。
- **编辑价值**: 刷 6302 全1「全解锁贴纸」(纯多人装饰、无战力，收集控向)。
- **风险/注意**: 低到中。94张里34张有解锁条件(任务/角色/徽章门控，如唯一缺的Key91需 ReqBadge=1352)：对60张无门控贴纸改6302=1风险最低；被门控的能否仅置1就真正可用并持久，取决于游戏是权威读取该标志(很可能)还是按门控重新推导(则可能被覆盖/再锁)，稳妥需同补对应徽章/任务/角色状态。6301排序改动安全但价值极低，做功能应以6302解锁为主。写后重算10段xxhash。

## 另记（低价值 / 存疑 / 勿追）

- 召唤石存档库存与装备 (1456-1460 / 1451/1454/1455)：当前配装页已能从真实 1000 槽库存中选择 4 个不重复实例，并可选写入全局装备槽 `1451`；这不等于每套配装都有独立召唤石字段。尚未落地的是新增或改造 `1456-1460` 库存实例（种类 hash/主特性/副参/双等级/rank）。`1452/1453` 验证显示登记表与库存存在耦合，新增实例还需安全插入 FlatBuffer 条目并验证游戏载入重算，因此继续留在实机待验边界。
- 珍玩/契约珍玩 Curios (1901-1904, +2002)：999槽珍玩库存，可在商店(妙具屋/Siero)鉴定开出物品/辉石/因子。工具在 save_parse.go 已定义 SaveID_CurioRewardItemID(1901)、SaveID_CurioIDs(2002) 常量却从未在任何功能里使用=作者预留但未实现。可做『塞满高价值珍玩』，但奖励是预掷存储的hash、新增需协调2002类型+奖励字段、且不确定游戏开启时是否重掷，又与工具已有的直接因子/物品编辑价值重叠。medium/risky。
- 角色 Fate / Master 永久强化（`1318` 位掩码 / `1323` 总 MSP）：当前配装只读链已经将 `1318` 低 11 位按已完成 Fate 篇章数联 `chara_status_fate`，并将 `1323` 联 `chara_master_exp` 与 `skillboard_unlock`、把属性解锁钳制到 Master Lv50。真实伊欧样本得到 Fate `HP +640 / ATK +165`、Master `HP +6000 / ATK +3000 / DmgCap +100`，再与 `1309/1310/1312/1313` 合成固定基准。这个结论只解决了配装读取与估算，不代表已经提供 Fate/Master“一键满”写入；强改进度的游戏合法性仍需实机验证。
- 编队方案命名预设 (2301-2303)：玩家在编成界面写入的命名编队(如『编队方案02』)，可编辑但纯便利型命名配置、非解锁/收藏/进度，价值低。
- 名片留言 (4604)：自己名片(unit 10600)的留言文本可写(工具已覆盖同块的4601推し角色)；纯外观、无解锁增益，锦上添花级。
- 【勿追·已被对抗验证否掉的伪收藏系统】1452/1453=召唤石按类型登记层(全置1不给实物，真库存在1456-1460)；2550/2551 与 2580/2581=实为任务清关记录(0x2xxxxx / 0x007X 命名空间，与工具已覆盖的2570/2571同族，非图鉴)；4201/4202=通用内部哈希字典(语义未明、疑内部事件/菜单标志)；5901/5902=召唤石按类型子层(与召唤功能重叠、且原判定字段归属错误)；6001/6002=教程『已看过』追踪(tutorial_menu，刷全1只会抑制教程弹窗、零或负收益)。这些看似『目录+标志』很像收藏，实为内部/进度状态，不建议做成功能。
- 【1202 勿做成改名】1202='TERMINAL'是内部常量(两份不同玩家存档字节相同)，非玩家名。真正的名字字段：玩家昵称=4413/834(见rank1)；剧情队长名『姬塔』=1101(UTF-16LE)与814(UTF-8)双副本、若做队长改名须同步改两处。
- 【工具向附赠】破译了 GBFR 的哈希算法 XXHash32Custom = 标准 xxh32(seed=0x0133ECF3，对 UTF-8 大写 Key)，EmptyHash=0x887AE0B0=xxh32("")。可把存档里的私有哈希键反解成游戏表 Key(交换所4001/贴纸/徽章的键定位皆可用)，对实现上述哈希表类功能有帮助。
