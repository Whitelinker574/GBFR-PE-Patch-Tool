# 配装模拟器 / 专精配置器 / 因子制造整合 —— 实现方案

> 由 29-agent 研究工作流产出，20 条结论经对抗验证、4 条被推翻纠正。数据源=游戏原表 sb.sqlite。

## 模拟器
v1 目标锁定用户原话「配好后算出因子加成总和（kannanote 那套）」——只做**因子→词条加成汇总表**，不做整机面板。这一层证据齐、可立即落地、零外部依赖。

算法（data-backed，怀疑论下最稳的一层）：
1) 从当前配装的 ≤12 颗因子逐颗取出「主词条(id,等级)+副词条(id,等级)」。注意现有 loadout 读取只给了因子物品 Level，没给主/副词条 id 与等级——须扩展读取（存档 TraitHashIDType/TraitLevelIDType @ 主/副 trait 单元，sigil_gen.go GetExistingSigils 已有同款读法可复用）。
2) 按词条 id 分组，把同名词条所有出现（主+副）的等级**相加**，封顶到该词条 maxLevel（traits.json.maxLevel == skill_status MAX(Level)）。
3) 用合并后等级查一次 trait_values.json[id][level]，按格式串占位符 {N}→LevelValue(N+1)、带%即百分比取出数值与单位。
4) 输出表：词条中文名 | 合并等级(是否封顶) | 数值 | 单位（flat/%）。多列词条按列语义各出一行。

后端建议加一个可测方法 LoadoutSimulate(sigils []{traitId,primLv,secLv}) []TraitBonus，写单测校验 ATK(SKILL_000_00 Lv50=2000 flat)/HP(Lv50=10000)/暴击(SKILL_003_00 Lv45=50%)三个锚点；前端在配装页实时重算、随因子勾选刷新。

诚实标注的局限（v1 不做或标「估算/待验证」）：
- 三个「联动」桶的**数值**贡献需要专精节点的 action_parts(Value1=每个量/Value2=封顶数)，这不在 skillboard_nodes.json（只有 desc 文本），desc 正则已被证明不可靠。v1 只按 GemCategory 显示 cat0/cat1/cat2+3 的因子数量供参考，联动数值标「待验证」或延后。
- 整机最终面板（白值ATK×(1+Σ%)、HP上限）需要 chara_status + 武器5张表（含 rebuild，叠加/替换未解、无实测校准）+ 专精 flat 节点，**未端到端验证**。v1 不出「最终面板数」，或出一个明确标注「估算」的框。用户诉求就是因子加成总和，这一块干净交付即可。
- 条件类词条（背水/浑身/连击加成）按「满层」显示并加开关标签。
- 召唤石、限界突破：v1 不计入。

核心待实测：同名词条「等级相加→封顶→查一次表」这条聚合规则结构上强烈成立但未对活存档面板实测，建议上线前用一套真实配装做一次游戏内校准。

## 专精配置器
现状只能「整套复制自该角色已有配装」（LoadoutEditor 的 masterySource 下拉）。可以升级成真正自由配置，且**不需要新数据**——data/skillboard_nodes.json 每个节点已带 char/cat/grp，grp 就是 4 个 rank 档：68DE92AC=R1、A96D9EBC=R2、4A5DDC7B=R3、3B99904D=EX（已验证映射，硬编码 4 个常量即可；skillboardNodeForHash 已返回 Grp）。

方案：
1) 后端加 MasteryNodePool(charaHex)：从嵌入的 skillboard_nodes.json 过滤 char==ownerCode，按 grp/cat 分组返回。各角色池子：R1=15、R2=27、R3=27、EX=30(主角PL0000/PL0100=42)——足够选。
2) 前端做 4 个 rank 标签页，每档一个横向节点网格，用户各选 10/10/10/20，实时计数；category(ATK/DEF/LIMIT) 在配额内**自由混搭**（已证无前置依赖、无连通性要求、无 category 内单类上限）。
3) 关键：必须给 validateLoadoutWrite 补配额校验。现有校验（loadout_write.go:479-499）只查「≤50 + 逐节点属本角色」，**不强制 10/10/10/20**，会写出游戏内可能非法的盘。按 grp 分组强制 R1=10/R2=10/R3=10/EX=20（满级唯一合法分布）。
4) 保留「复制现有」作为快速起手/种子。可选：离线从 skillboard_auto_acquire 提取 29角色×3套官方预设(偏ATK/DEF/LIMIT) 成 data/mastery_presets.json 作一键载入模板。

合法性能自由校验：约束简单且被四路证据锁定（unlock 表钉死 10/10/10/20、rank_adjust、auto_acquire、SaveData2 四套真实build），配置器自己加配额判定即可，不用靠复制。

诚实标注：非 10/10/10/20 分布是否被游戏拒绝未实测→默认强制满配额；写入只落 3007 预设槽，用户须在游戏里「读取该配装」才生效（与现有机制一致）；group 的 Unk2/3/4 语义与「50级后额外5次强化」未解，但都不影响盘本身合法性。

## 因子制造整合
技术已实证可通，推荐 **UI 层串联复用两套已验证 API，不新造存档接口**：
第1步 SigilGen.ApplyQueue(outputPath) 把队列造进存档并落盘，返回 OutputPath；
第2步以该 OutputPath 调 App.LoadoutEditContext(outputPath, charaHash) 重读——新造的**通用**因子会自动出现在 ctx.sigils 池（gemBySlotID 收所有非空因子，通用因子过安全闸门）；用户勾选后 App.LoadoutApply 把 SlotID 写进 1403。存档本就允许配装引用 2706=EmptyHash 的库存因子，造出的记录与游戏自产库存因子无法区分。

工程串接点：
- ApplyQueue 目前只返回 {CreatedCount,VerifiedCount,OutputPath}，**没有 gemUnitID→newSlotID 映射**。要在配装池里自动高亮「刚造的因子」，给 ApplyResult 补一个 NewSlotIDs 字段（firstNewSlotID..firstNewSlotID+n-1，代码里已有这两个量），小改动、低风险。
- 可选一键版：加一个薄后端方法，在单个 SaveData 缓冲内先 PatchSigil 造因子、再 applyResolvedWrite 写配装、最后一次 FixChecksums/Write，省掉中间落盘重载。

必须处理的坑（来自已验证结论）：
1) 角色专属因子被配装安全闸门拒绝（isCharacterSigil && !precedent），只有**通用因子**能直接装。v1 只支持通用因子制造入装（正好覆盖主场景：造你没有的meta通用因子）；要支持角色专属需放宽 loadout_write 白名单或造时把该角色补进 precedent。
2) 每造一颗永久占空槽、2701 只增不减，批量制造膨胀库存（GetExistingSigils 有 500 条渲染上限信号）——UI 提示。
3) 两步跨文件跨对象是最大摩擦点，不重载配装侧就看不到新因子（可控）。
4) flags=2 vs 3 跨档差异，纯显示、不阻塞。
5) 造因子强度预览缺每级数值——正好复用本次要生成的 trait_values.json（与模拟器同源），顺带补上 traits.json 缺的逐级数值。

## UI 布局重设计
现状配装区被塞在窄单列里（CharaStats 类 max-width:720、LoadoutEditor 单列纵向堆叠、因子/技能是 max-height 纵向滚动小列表）。按用户诉求「做大、往右占更多空间、15槽横向图标网格、选中后大空间自定义」重设计为**全宽双栏工作台**：

- 顶部/上方：15 个预设槽做成**横向响应式图标 chip 网格**（现有 grid-template-columns:repeat(auto-fill,minmax(38px,1fr)) 已是雏形，扩成占满整行、区分占用/空槽、显示配装小名）。点槽在主区打开编辑器。
- 左窄栏：角色列表。
- 右主区（宽）：多列布局。左列=武器(下拉)+技能(≤4)+专精入口；中/右列=12 个因子槽做成**横向图标磁贴网格**（替代当前纵向滚动列表），点磁贴内联展开该因子编辑；最右**常驻模拟器面板**显示「因子加成总和」表，随勾选实时刷新。
- 专精单独一个可展开面板/弹层：4 个 rank 标签页 + 配额计数器 + category 自由混搭网格。
- 造因子入口就近放在因子磁贴区旁（「没有想要的因子？去制造」→ 造完回到本池自动高亮）。
- 用相对单位 + grid/flex + container query（代码库已用 container-type），沿用羊皮纸/gold 主题变量（var(--gold)/--line-gold 等），保持与现有组件一致。

## 需先做的数据工作
 - data/trait_values.json（新建，离线生成后 go:embed）：由 sb.sqlite/skill_status(Key+Level+LevelValue1..10) join csm.json(TXT_SKILL_EXPLAIN_* 格式串) 展平。每词条含 id、中文名、maxLevel、每列语义与单位（占位符{N}→LevelValue(N+1)，带%即百分比）、Lv1..maxLevel 逐级数值。这是模拟器地基，也供造因子强度预览。165/166 覆盖，标注 SKILL_166_50 缺失、多列词条(暴君主值在LevelValue2 等)需按格式串逐条建列→语义映射。
 - 扩展 LoadoutPickSigil 及配装因子读取：补 primaryTraitId+level、secondaryTraitId+level（从存档 TraitHash/TraitLevel 单元读，复用 sigil_gen.go GetExistingSigils 的读法）。这是代码改动但属数据形状变更——模拟器要靠它知道每颗装备因子贡献哪个词条、哪一级。
 - 专精无需新数据：skillboard_nodes.json 已含 char/cat/grp。可选生成 data/mastery_presets.json（离线从 skillboard_auto_acquire 提 29角色×3套官方预设）作配置器快速起手模板。
 - （延后，仅当要做整机面板才需要）chara_status.json、武器 base/awake/plus/rebuild/level_sync 各表——标注 rebuild 叠加/替换未解、需游戏内实测校准，v1 不做。

## 实现步骤
 - 离线生成 data/trait_values.json（skill_status join csm.json）并 go:embed；写加载器 + 单测锚定 ATK Lv50=2000 / HP Lv50=10000 / 暴击 Lv45=50%。

 - 后端加 MasteryNodePool(charaHex) 按 rank/cat 分组；给 validateLoadoutWrite 补 10/10/10/20 配额校验；保留复制现有作种子。
 - 前端做专精配置器（4 rank 标签页 + 计数器 + category 自由混搭），替换/增强现有仅复制的下拉。
 - 造因子整合：给 ApplyResult 补 NewSlotIDs；UI 串「造→重读入池→自动高亮→写进配装」，v1 仅通用因子。可选做单缓冲一键方法。
 - （延后）整机面板：chara_status + 武器多表，先解决 rebuild 叠加/替换并做一次游戏内面板实测校准，再以明确「估算」标签出面板。

## 风险
 - 模拟器核心假设「同名词条等级相加→封顶→查一次表」未对活存档游戏内面板实测——上线前须做一次端到端校准，否则数值可能与游戏不符。
 - 联动数值需专精 action_parts 的 Value1/Value2，不在嵌入 json 里；desc 正则已证不可靠→v1 只显示因子分类计数、联动数值延后或标待验证。
 - 整机最终面板涉及武器5张表（rebuild 叠加/替换未解、且是最大属性来源）+召唤石+限界突破，均未验证→v1 不出最终面板数或明确标估算。
 - 专精配置器：非 10/10/10/20 是否被游戏拒绝未实测→默认强制满配额；写入仅落 3007 预设槽，须游戏内读取配装才生效。
 - 造因子：角色专属因子被配装安全闸门拒绝（v1 仅通用因子）；2701 单调只增、批量造膨胀库存（500 条渲染上限）；两步跨文件是最大 UX 摩擦点。
 - 多列词条需逐条建「列→语义→单位」映射（如暴君主值在 LevelValue2 而非 1），须靠 csm.json 格式串人工核对少数条目。
 - DLC 词条在 skill_status 以 hash 为 Key，须确认 gem.SkillId 为 hash 时 join 覆盖全部可编辑因子（基础游戏字符串 Key 已验证）；SKILL_166_50 在 skill_status 缺失需兜底。

# 已验证的关键事实

## 已验证结论
- [sim-data] 值表就是 sb.sqlite 的 skill_status 表：关键列 Key(词条ID) + Level(词条等级,1起) + LevelValue1..LevelValue10(该级的数值)，另有 LevelDescription(指向 csm.json 的说明文本)。它是「词条×等级→数值」的地基表。（证据：sb.sqlite 表 skill_status 6320 行；headers 定义 add_column|Key|hash_string、LevelDescription|hash_string、Level|uint、LevelValue1..10|float。样本 Key='SKILL_000_00' 有 Level 1..50 共 50 行。）
- [sim-data] 链路：sigils.json.primaryTraitId(如 SKILL_000_00) == traits.json.internalId == skill_status.Key(明文)。traits.json.hash(0x50079A1C) 就是 GBFR 私有 hash= xxh32(seed=0x0133ECF3, 大写UTF-8) 作用于 internalId 的结果，实测吻合。因此 Go 侧直接用 primaryTraitId 字符串 join skill_status.Key 即可取到整条等级曲线。（证据：python 实测 gbfr('SKILL_000_00')=0x50079A1C == traits.json 中 SKILL_000_00 的 hash；EmptyHash 校验 gbfr('')=0x887AE0B0 吻合。traits.json 每条 internalId=SKILL_xxx_00、hash=对应值。165/166 个 trait 的 internalId 能在 skill_status.Key 命中。）
- [sim-data] 存档里每个因子把「词条等级」独立存成一个字段(TraitLevelIDType)，与因子物品等级(GemLevelIDType,1..15)分开存；主/副词条各一份，位置= TraitSlotBase + (gemIndex*100) 和 +1。这个存档里的 TraitLevel 就是直接 1:1 索引 skill_status.Level 的数字。（证据：sigil_gen.go:526-548、789-817：primaryTraitUnit=TraitSlotBase+gemIndex*100，secondaryTraitUnit=+1，分别读 TraitHashIDType/TraitLevelIDType；GetExistingSigils 返回 Level/PrimaryLevel/SecondaryLevel 三个独立值。）
- [sim-data] 数值单位(百分比 vs 绝对值)由 LevelDescription 的格式串决定，不在数字本身：占位符 {N} 映射 LevelValue(N+1)，{N} 后带 % 即百分比、不带即绝对值。两类都存在——攻击力 SKILL_000_00 是绝对值(攻击力+{0}，Lv15=40,Lv50=2000)，体力是绝对HP(最大HP+{0},Lv50=10000)，暴击率是百分比(暴击率+{0:.1f}%,Lv45=50)。（证据：csm.json：TXT_SKILL_EXPLAIN_000_00='攻击力+{0}'、_001_00='最大HP+{0}'、_003_00='暴击率+{0:.1f}%'；对应 skill_status 数值 ATK Lv1=4/Lv50=2000、HP Lv1=200/Lv50=10000、Crit Lv1=5/Lv45=50。）
- [sim-data] 关键坑：多值词条的「主数值」不一定在 LevelValue1，必须按格式串占位符取对应列。例：暴君 Tyranny SKILL_027：攻击力%在 LevelValue2({1})，随等级 5→50 增长；LevelValue1({0}=最大HP-%)恒为20。浑身/伤害上限/协同攻击等都是多列(cols=2/3/6)。（证据：skill_status SKILL_027_00: Lv1 V1=20,V2=5 → Lv30 V1=20,V2=50；csm fmt='攻击力+{1:.1f}%\n最大HP-{0:.1f}%'。SKILL_020 DMG Cap cols=3(普攻/能力/奥义各一列)，SKILL_009 协同攻击 cols=6。）
- [sim-data] 常见输出词条样本(skill_status,数字=LevelValue,取 Lv1/Lv15/Lv50或该词条上限)：攻击力SKILL_000_00 4/40/2000(绝对,max50); 体力SKILL_001_00 200/2600/10000(绝对HP,max50); 暴击率SKILL_003_00 5/19/50%(max45); 暴击伤害SKILL_013_00 5/35/50%(max30); 伤害上限SKILL_020_00 3/45/250%(max65,3列); 背水SKILL_005_00 10/70/100%(max30); 浑身SKILL_006_00 8/50/70%(max30,2列HP100%/25%); 连击加成SKILL_024_00 10/70/100%(max30); 追击SKILL_151_00(max45,2列)。（证据：scratchpad/trait_coverage.txt 第2-166行逐条列出 skid/中文名/max/cols/Lv1/Lv15/LvMax/格式串，由 skill_status join csm.json 生成。）
- [sim-data] 角色专属「终结/觉醒」类词条(SKILL_114..172 的 _00/_01/_02/_03 三或四段)在 skill_status 里 Lv1..14 多为 0，真正数值只在 Lv15 出现，属定值效果曲线；纯抗性类(中毒/灼热等)统一 10/70/100% 走 max30。（证据：trait_coverage.txt：SKILL_114_01 Fearless Spirit Lv1=0/Lv15=10；SKILL_046_00 中毒抗性 Lv1=10/Lv15=70/Lv30=100 fmt='受到的伤害-{0}%'。）
- [sim-formula] 因子/词条的每级数值就在游戏表 skill_status（6320行），这正是 traits.json 缺的『每级数值』。列：Key(词条ID)、Level、LevelValue1..10、LevelDescription。Key 对基础游戏词条是字符串直存（如 'SKILL_000_00'），对 DLC 词条是 hash。gem.SkillId1/SkillId2 直接 join 到它。（证据：sb.sqlite skill_status 行: Key='SKILL_000_00' Level=50 LevelValue1=2000.0 LevelDescription='TXT_SKILL_EXPLAIN_000_00'; gem 表列 SkillId1='SKILL_000_00'/SkillId2(副词条)）
- [sim-formula] 同名词条叠加规则=把所有同名因子的等级相加，封顶到该词条的 maxLevel，再用合并等级去 skill_status 查一次值（不是各因子独立算再相加）。maxLevel 就是硬上限，且每词条不同：直方图 125个词条封顶15、51个封顶30、37个封顶50、15个封顶45、1个封顶65(伤害上限)。（证据：traits.json 的 maxLevel(ATK)=50/HP=50/暴击率=45/昏厥=45 与 skill_status 对应 Key 的 MAX(Level) 完全一致；单因子单词条等级≤15(gem_rare rarity4 MaxedLevel=10) 远小于50，故到50必须靠叠加）

## 关键数据表
- skill_status — 词条数值主表：Key(词条ID,明文SKILL_xxx_00)+Level(1..max)+LevelValue1..LevelValue10(float,每级数值)+LevelDescription(→csm.json说明文本)。6320行,165/166词条全覆盖
- skill — 词条/技能目录：Key(SKILL_xxx_00)+Name+Summary+GemCategory,261行,是 skill_status 的名称侧
- gem — 因子(sigil)目录：SkillId1/SkillId2(→skill.Key)+Key(GEEN_xxx)+Name+Rarity+Category,1034行,连接 因子→词条
- status — 168行状态/buff定义(StatusName/Category/HasLevels等),用于状态类词条的展示,非数值曲线
- csm.json(非sqlite) — 简中文本：TXT_SKILL_xxx=词条名, TXT_SKILL_EXPLAIN_xxx=带{0}{1}占位符和%的说明串,决定数值单位与列语义
- skill_status(6320行)：每个词条逐等级数值。Key=词条ID(基础游戏为SKILL_xxx_yy字符串/DLC为hash)、Level、LevelValue1..10、LevelDescription。这是 traits.json 缺的每级数值的权威源
- skill(261行)：词条主表。Key、Name(→csm)、GemCategory(0基础/1攻击/2防御/3支援/4特殊)、IsResistance
- gem(1034行)：因子定义。SkillId1(主词条)、SkillId2(副词条)、Category、Rarity、Name。因子→词条的桥
- gem_rare(5行)：稀有度→DefaultLevel/MaxedLevel(rarity4=默认7/满10)
- chara_status(589行)：角色基础面板逐级。Key(PLxxxx)、Level、Hp、Attack、DefenseLikely、Stun、CritRate
- weapon(410行)：武器定义。Key(WEP_PLxxxx_yy)、WeaponStatusAwakeId、WeaponStatusPlusId、MaxLevel、LastAwakeningLevel、Name、CharaId
- weapon_status(2602行)：武器基础面板逐级 Attack/Hp/StunPower/CritRate(Level1..150)
- weapon_status_awake(580行)：觉醒加成,按WeaponStatusAwakeId+觉醒等级1-10给 flat AttackGain/Hp
- weapon_status_plus(99行)：强化『+』加成,按WeaponStatusPlusId+等级
- skillboard_layout(2895行)：专精节点布局。Key=存档3007里的节点hash、SkillboardEffectOrUiId、SkillboardCategoryId(SB_ATK/SB_DEF/SB_LIMIT)、CharacterId
- skillboard_effect(2895行)：节点→动作部件。Key、SkillboardEffectActionPartsId1..3、名/说明文本ID
- skillboard_effect_action_parts(2915行)：专精节点强类型数值。Value1..10、StatusId、AbilityId、MainType、SubType、Conditional、ConditionBehavior(联动数量型节点用Conditional!=0编码)
- summon_base_param(23行)：召唤石逐级参数 ParamID(400ATK/401HP/402暴击/403昏厥...)L1..L10——但对面板的被动贡献未证实
- limit_bonus_param(1123行)：限界突破逐级 flat ATK/HP/暴击(LBP_EFF_ATK01等)——真实面板有贡献但不在四件套范围
- traits.json(D:\gbf\...\data\traits.json,166词条)：有 internalId/hash/displayName(英)/maxLevel,maxLevel==skill_status硬上限,但无每级数值(须配 skill_status)
- skillboard_nodes.json(D:\gbf\...\data\skillboard_nodes.json,2895节点)：desc已填好数值,~1731无条件可正则,174个含『装备数量』的联动节点
- skillboard_layout (2895行) — 专精节点主表。列: Key(节点hash=skillboard_nodes.json的hash), SkillboardCategoryId(SB_ATK/DEF/LIMIT), SkillboardGroupId(=四个rank档), CharacterId(PLxxxx), 另有40个Bool列+SkillboardEffectOrUiId。每category各965行, 29角色
- skillboard_group (4行) — 四个group即四个rank档: 68DE92AC/A96D9EBC/4A5DDC7B/3B99904D。列 Key,Unk2,Unk3,Unk4(=10/10/1, 10/20/3, 10/30/5, 20/30/5，疑为点数阈值/技能激活等级)
- skillboard_unlock (50行) — MasterLevel 1..50。列 Rank1/Rank2/Rank3/RankEXNodeCountAdd(累计=10/10/10/20=50节点槽), MasterLevel, HealthAdd/AttackAdd/DmgCapAdd(升级被动数值)
- skillboard_category (4行) — SB_DEF/SB_ATK/SB_LIMIT + 未用的544087E7(layout里0行引用)。列 Unk1,Unk2,Unk3,Key,Unk5 = 3,6,6,3
- chara_power_skillboard_rank_adjust (3行) — 68DE92AC→rank1(×1.0), A96D9EBC→rank2(×2.0), 4A5DDC7B→rank3(×3.0)。3B99904D(EX)不在表中=无rank数值加成
- skillboard_auto_acquire (1450行=29角色×50) — 每角色3列(Id1/Id2/Id3)=3套预设build(偏ATK/偏DEF/偏LIMIT), 每套精确10/10/10/20, 按rank顺序排列。是现成的合法build样本
- loadout preset 3007向量(UnitID 20000-20614)与party loadout(104000-104003) — 存档里唯一存放专精节点hash的位置, 定长50槽
- 存档 FlatBuffer 单元表(非游戏DB)——因子槽 gemUnitID≥30000：IDType 2702=因子SlotID(配装1403引用的就是它)、2703=因子hash、2704=等级、2706=装备角色(EmptyHash=库存/角色hash=已装)、2707=flags；全局 2701=MaxSlotID 计数器

- 项目数据文件：data/sigils.json(因子→主特性/可选副特性/等级)、data/traits.json(词条名/hash/maxLevel，无逐级数值)、data/secondary-trait-rules.json、data/skillboard_nodes.json(专精节点翻译)
