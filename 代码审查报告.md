## 一、高危问题（可能崩溃 / 损坏）

### A-1【high】App 结构体无锁，Wails 并发调用下存在数据竞争与 use-after-close
- 位置：`app.go:117-145`（`hProcess`、`moduleBase`、各 hook/cave 地址、`damageMeterView` 均为无保护共享状态）
- Wails 对每个前端调用起独立 goroutine。前端轮询 `DamageMeterGetStatus`（`app.go:2172`，直接解引用 `a.damageMeterView` 裸指针）与 `CharaDetach`/`ensureGameProcess`（`app.go:1099-1101`，`CloseHandle` 后清零句柄）并发时，可能对已关闭句柄读写内存或解引用已清零指针，导致工具崩溃。
- 仅 `damageOverlay` 和存档快照（`saveSnapshotMu`）有锁，进程/hook 相关字段完全裸奔。
- 附带竞态：`ensureDamageMeter`（`app.go:2191-2211`）无锁 check-then-act，并发时泄漏映射句柄+视图。

### A-2【high】PatchSigil 用 panic 做错误处理，且副词条槽未预校验
- 位置：`sigil_store.go:335`（内部 8 处 `must()`，`must` 在 `sigil_store.go:605` 直接 panic）；调用方 `sigil_gen.go:524-552 ApplyQueue`
- `ApplyQueue` 预校验只查了主词条槽，没查副词条槽。当 `hasSecondary==true` 而副词条条目缺失时，写档中途直接 panic（Wails 绑定方法内崩溃而非返回错误）。存档写入路径用 panic 做控制流是脆弱设计。

### ~~A-3【high · 半成品】上限突破的"保存"是空实现~~ → 已推翻，降级为 C-4 清理项
- **更正（2026-07-16，据用户实机确认"上限突破会直接保存" + CE 表交叉核对）**：先前判断有误。上限突破的持久化机制与因子/召唤石不同：`OverLimitSetSlot` 直接把属性/等级/effectID/数值写进游戏的"当前选中突破槽"内存（`SelectedAddr + i*0x10`），游戏在正常保存流程中会把这块结构落盘，**无需工具主动调用存盘函数**。
- 因此 `OverLimitCommit`（`overlimit.go:239`）只读状态、不做持久化是**符合设计的正确行为**，不是 bug。
- 而 `findOverLimitCommitAddr`（`overlimit.go:358`）+ `overLimitCommitPattern/Mask` + overlimit 场景的 `callRemoteOneArg`——是"远程调用游戏存盘函数"这条**被放弃的旧方案**的残留，因为发现游戏会直接落盘所以没接线。**属死代码，应删除（并入 C-4），不是待修 bug。**
- 数据核对：`overLimitAttributeOptions` 的 11 个属性 ID 与 CE 表"上限突破 ID"列表（C4925BD7 攻击力 / 43B7581D 普通攻击上限 / 9C555433 能力伤害上限 / 4A4C093D 奥义伤害上限 等）逐一吻合；Go 的"基础 ID + 独立等级位掩码"表示法与 CE hook `NBGFR091_int` 的"4 词条 ID + rank"结构一致。**无数据错误。**

## 二、中危问题（功能缺陷 / 泄漏 / 潜伏 bug）

### B-1【medium】overlimit hook 无卸载路径，CharaDetach 不恢复
- `OverLimitEnable`（`overlimit.go:153-189`）向游戏进程写入 E9 跳转 + code cave，但全项目没有恢复函数；`CharaDetach`（`app.go:1097-1098`）只恢复了 sigil/currency 两个 hook。代码注释自述"必须在进程还在时恢复 hook"，与实现矛盾。detach 后游戏进程残留指向工具 cave 的跳转。

### B-2【medium】两套独立的存档解析实现并存
- `save_parse.go`（规范 FlatBuffers vtable 读取器，只读摘要用）vs `sigil_store.go`（启发式暴力字节扫描器 `findUnit`，所有写入用）。
- 重复常量：`XXHash64SaveSeed`(save_parse.go:13) ≡ `SaveHashSeed`(sigil_store.go:28)；`hashSectionInfos`(save_parse.go:663) ≡ `hashSections`(sigil_store.go:33)。
- `findUnit` 的 step=1 非对齐二次扫描理论上可能命中假条目写错位置（有写后回读校验兜底）。典型失忆产物：第二次实现时忘了第一套的存在。

### B-3【medium】parseFloatUnits 浮点解析错误（潜伏 bug）
- `save_parse.go:435`：`float32(r.u32(...))` 按数值转换而非 IEEE-754 位模式重解释，正确写法是 `math.Float32frombits`（`overlimit.go:344`、`app.go:2043` 用法正确）。FloatTable 目前无消费方所以未爆发，一旦将来读取即得错值。

### B-4【medium】注入的临时 DLL 永不清理、永不卸载
- `app.go:2495`：每次注入落盘一个 `patch_core_<纳秒>.dll`，从不删除（被目标进程锁定），磁盘持续累积；`app.go:2502-2526 injectDLL` 只 LoadLibrary 不 FreeLibrary，反复开关在目标进程内单调累积 DLL 实例。
- 配套：C++ 侧所有 hook 的 code cave（`dllmain.cpp:161-189 AllocNear`）分配后无人记录地址，Go 侧 `restoreMonsterEnhance`（`app.go:2393-2432`）只写回原字节，cave 每开关一轮泄漏一块。

### B-5【medium】无存档文件时所有运行时写入功能被拒绝（需确认是否有意）
- `save_backup.go:389 snapshotBeforeLiveSaveChange` 用 `allowEmpty=false`：默认存档目录没有 SaveData1-3.dat 时，角色次数、货币、药水、上限突破、召唤石、因子、全称号等**全部实时写入被"未找到存档"拦下**。防呆略过严，需确认设计意图。

### B-6【medium】unsafe.Pointer 违规用法（go vet 告警）
- `app.go:2176/2177/2186`：`uintptr` 字段再转 `unsafe.Pointer` 解引用。映射地址稳定所以实际能用，但属 vet 明确告警的模式。

### B-11【medium·据 CE 表逐字节交叉验证新增】游戏内保存函数裸 RVA 0x79D820 无校验即远程调用
- 位置：`sigil_memory.go:11/358 sigilMemorySaveRVA` 与 `summon_memory.go:17/269 summonSaveFunctionRVA`（同为 0x79D820）
- `saveSigilMemory`/召唤石保存直接 `callRemoteOneArg(moduleBase+0x79D820, ...)` 把该地址当函数远程线程调用，**没有任何函数序言字节校验**（对比工具其他补丁点都有 expected-bytes 前置校验）。
- 危害等级高于普通裸 RVA：读错地址只是拿到垃圾值，而**调用错地址=在游戏进程执行任意代码路径**，可能崩溃或写坏存档。游戏版本更新后该偏移最易失效。
- 建议：调用前读取该地址前若干字节做函数序言/特征校验，不匹配则拒绝调用并提示"当前游戏版本不受支持"。
- 交叉验证背景：CE 表（NidasBot 0.8/0.8.1）中对应的保存/生成逻辑是混淆的编译 Lua 字节码，无法互证此地址，故只能靠工具自身加校验兜底。

### B-7【medium】预编译 patch_core.dll 与源码无同步保障
- `build/bin/patch_core.dll` 是提交的预编译产物（经 `//go:embed` 编入 exe），构建脚本不重编 C++，改了 `src_dll/*.cpp` 忘编译就会源码与二进制脱节——对失忆 agent 是高发事故。建议构建脚本加校验或强制重编。

### D-1【medium·已人工确证】sigil_ctdata.go 因子名表存在错误映射
- 工具的两套因子数据自相矛盾：显示名表 `sigil_ctdata.go`/`_en.go` 与工具自己的 `data/sigils.json` 对同一 hash 给出不同名称。已确证 2 例（sigils.json 与外部表一致，判 ctdata 错）：
  - `0x9C2399DA`：ctdata=「奋不顾身IV+ / Life on the Line IV+」，sigils.json=「Aegis V+（守护）」
  - `0xEDA2116A`：ctdata=「凝神III+ / Nimble Defense III+」，sigils.json=「Steel Nerves V+（坚持）」
- 另 2 例（`0x72466F3C`、`0x77802303`）仅外部表支持（sigils.json 无收录），疑同批错标，待核。
- 影响：存档里出现这些因子时，界面显示错误名称（不影响生成合法性，属显示层）。建议以 sigils.json 为准批量校正名表。

### D-2【medium·已人工确证】weapons.json 武器中文名成对互换（4 对 8 条）
- 每条武器 nameCn 前缀方括号内即其英文名的对应词，但后缀名在成对武器间被互换：
  - `0FAE326F` Ethereal Lasher ↔ `2A924378` Erinnerung（"回忆/灵驭"换反）
  - `209AA176` Blanc Comme Neige ↔ `72AAE787` Knight of Ice（"如雪/冰霜骑士"换反）
  - （另据外部表）`9712FDF2` Antwerp ↔ `98F8CA3E` Lord of Flames；`B70179F1` Crimson Finger ↔ `F4C1DE98` Golden Fists of Ura
- 另有 7 把武器 nameCn 为空（米迦勒之剑、以弗所、风之宝剑等），可从权威表直接补齐。

### D-3【low·已确证】summon_base_params.json 损坏——但是孤儿文件，无功能影响
- 文件首字节 0xb3 非法、整体非合法 JSON。**但 Go 侧只 `//go:embed` 了 summon_skills.json 与 summon_sub_params.json，从未加载 base_params**，故损坏不影响任何功能。属清理项：删除或重新导出。

### D-4【medium】DLC 内容覆盖不全（工具停留在游戏 2.0.2，晚于最新 DLC）
- `weapons.json`/`items.json` 标注 version 2.0.2，权威表为 V4/7.13。两轮审计一致：因子缺 ~205–241 条（含圣德芬 DLC 天司长系列、天星系列各档位、空白"因子强化/补充"、各种 I+/II+ 低档），因子特性缺 ~40、祝福特性缺 ~64（含抗性类与 DLC）。
- 影响：编辑器里选不到这些较新/DLC 因子与特性。与 README 自述"DLC 无尽黄昏更新后可能有部分功能失效"一致，属"未随 DLC 更新"，非失忆丢数据。若要支持最新 DLC 需批量补数据。

### D-5【low】其余核对结论
- **完全一致（无问题）**：`items.json`（312 条 100% 对上，本就同源）、`summon_sub_params.json`（22 条双射）、`characterNameByHash`（29 角色 hash 全一致，反倒是外部角色表自己 CN 列错位）。
- **召唤石 Blizzadile II**：外部表 hash 残缺（Excel 丢字符成 `95420`），工具 `0x03C95420` 应无误；但工具把黑龙伊弗欧系列(3 条)与 Lurker in the Dark 系列(3 条)**都命名为"永暗藏匿者"**，导致多个重复名，建议区分。
- 上限突破属性：工具用"基础 ID + 等级位掩码"参数化（11 条），外部表按 I–V 档位各列独立 hash（22 条）。**需核实**：若游戏内高档位（II–V）确实使用独立 hash 而非位掩码，则工具只登记 I 档 hash 可能无法识别/写入高档位——建议实测确认。

## 五、修复优先级建议

1. **A-1 并发锁**（唯一会让工具自身崩溃的问题）+ A-2 写档 panic 路径
2. **B-11 保存函数裸 RVA 0x79D820 加校验**（调错地址=坏档，风险最高的裸 RVA）
3. B-1 overlimit hook 卸载、B-8/B-9 前端交互 bug、B-3 一行修复
4. **D-1/D-2 已确证的数据错误**（因子名表错映射、武器名互换）——纯数据修正，低风险高收益
5. B-4 临时 DLL 清理、B-7 DLL 构建校验、C-1/C-2 构建体验
6. B-10 i18n 补译与冲突、C-3 文档更新、C-4 死代码清理（含 A-3 更正后的 overlimit commit 死代码、损坏的 D-3 孤儿文件、坏死的 sigilMemorySelectedPattern）
7. D-4 DLC 数据补全（工作量大，视是否要支持最新 DLC 决定）

> 注：A-3（上限突破"保存"）经用户实机确认 + CE 表交叉核对**已推翻**，`OverLimitCommit` 空实现是正确设计，相关 `findOverLimitCommitAddr` 属应删死代码，不再是待修项。
